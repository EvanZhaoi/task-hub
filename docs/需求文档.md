## 1. 背景与目标

### 1.1 背景
- 公司内部有大量开发任务，但开发资源**有限且分散**在各个部门
- 已有**虚拟货币制度**（≈ 真实奖金，关系员工收益）
- 缺乏市场化的资源调配机制，导致部分任务长期无法推进

### 1.2 目标
搭建一个**内部任务交易撮合平台**，用市场化的方式让"谁有活"和"谁有空"自然对接。

### 1.3 MVP 范围声明

| 类别 | MVP 做 | 留到二期 |
|---|---|---|
| 任务流转 | 发布、投标、选标、指名、完成确认 | — |
| 资金 | ❌ 不做（走内部系统） | — |
| 信用 | — | ✅ 信用评分 |
| 匹配 | — | ✅ 任务分类 + 技能匹配 + 推送 |
| 定价 | — | ✅ 历史成交均价 |
| 仲裁 | — | ✅ 平台介入机制 |
| 用户体系 | ✅ 对接外部接口 | — |
| 附件 | ✅ 文件上传 | — |
| 任务描述 | ✅ 富文本（WYSIWYG） | — |
| 搜索 | ✅ 关键词搜索（任务标题 + 描述） | 全文搜索 / 高级搜索语法 |
| 分页 | ✅ 任务大厅列表分页 | 老板视图也分页（暂以筛选代替） |
| 老板视图 | ✅ 全局甘特图 + 按付款账号筛选 | — |
| 个人视图 | ✅ 个人时间线（甘特图） | 高级数据汇总 |

## 2. 角色与人员体系

### 2.1 系统内角色

| 角色 | 描述 | 系统内权限 |
|---|---|---|
| **任务发布者** | 任何员工 | 浏览、发布、投标、确认完成、取消 |
| **接单开发者** | 任何员工 | 浏览、投标、协商、交付 |
| **老板** | 拥有全局视图权限 | 看所有任务的甘特图视图 |

> MVP 不区分"管理员"角色。付款账号的管理在**公司内部系统**进行，TaskHub 只引用账号 ID。

### 2.2 人员信息来源（关键决策）

**TaskHub 不维护本地用户表**，人员信息全部从**外部人员信息接口**获取：

- **正式引用字段**：统一保存公司工号，工号在公司内唯一且不会变更
- **历史展示信息**：姓名、部门等在任务发生时通过 JSON 快照保存
- **不存本地用户表**：不建立人员外键，不新增 `user_source`
- **缓存策略**：在 TaskHub 内部做短期缓存（建议 1 小时），减少对外部接口的调用
- **接口约定**（待补）：见 §13 遗留问题 L6

### 2.3 权限模型要点

- 权限与外部接口中的"部门 / 角色 / 职级"绑定
- MVP 阶段简化为：**所有已登录用户权限对等**，老板视图单独入口
- 二期可引入：部门隔离、职级限制等

## 3. 核心概念

| 概念 | 描述 |
|---|---|
| **任务（Task）** | 一笔开发工作的最小单位，从发布到完结有明确状态流转 |
| **投标（Bid）** | 开发者对应任务的报价响应 |
| **协作组（Collaboration Group）** | 多人协作投标时的一组开发者 |
| **付款账号（Payment Account）** | 标记任务的资金来源，平台只引用 ID |
| **附件（Attachment）** | 任务相关的文件（PRD、UI 稿、接口文档等） |
| **状态变更记录（Change Log）** | 任务进行中所有状态/字段变更的审计日志 |

## 4. 任务状态机

```
   ┌────────┐
   │ DRAFT  │  (草稿，发布者编辑中)
   └───┬────┘
       │ 发布
       ▼
   ┌────────┐
   │  OPEN  │  (招标中)
   └───┬────┘
       │ 超时无投标              │ 选标
       ▼                        ▼
   ┌────────┐              ┌──────────┐
   │ FAILED │              │ ASSIGNED │  (进行中)
   └───┬────┘              └────┬─────┘
       │ 复制新建                │ 发布者确认完成
       │                        ▼
       │                   ┌──────────┐
       └──────────────────►│COMPLETED │
                           └──────────┘

   DRAFT / OPEN / ASSIGNED → CANCELLED  (取消)
```

### 4.1 状态定义

| 状态 | 含义 | 终态？ |
|---|---|---|
| DRAFT | 草稿，未对外可见 | ❌ |
| OPEN | 招标中，对全员可见 | ❌ |
| ASSIGNED | 已选标，进行中 | ❌ |
| COMPLETED | 已完成 | ✅ |
| FAILED | 流标 | ❌（可复制新建） |
| CANCELLED | 已取消 | ✅ |

### 4.2 状态转换规则

| 起始状态 | 操作 | 结果状态 | 触发者 |
|---|---|---|---|
| DRAFT | 发布 | OPEN | 发布者 |
| DRAFT | 取消编辑 | CANCELLED | 发布者 |
| OPEN | 选标 | ASSIGNED | 发布者 |
| OPEN | 流标（超时无投标） | FAILED | 系统/定时 |
| OPEN | 撤回 | CANCELLED | 发布者 |
| ASSIGNED | 确认完成 | COMPLETED | 发布者 |
| ASSIGNED | 双方同意取消 | CANCELLED | 发布者+中标者 |
| FAILED | 复制新建 | DRAFT（新建一条） | 发布者 |
| * | 字段变更 | 状态不变，生成 Change Log | 双方 |

## 5. 核心流程

### 5.1 流程 A：任务发布（招标模式）

1. 发布者填写任务信息：标题、**富文本描述**、**付款账号**、预算金额、期望交期、招标截止时间
2. **发布者线下与付款账号管理员确认金额**（平台不拦截）
3. 点击"发布" → 状态变 `OPEN`，对全员可见

> 金额一旦发布后变更，需发布者**线下再次找管理员确认**。

#### 5.1.1 富文本描述规范
任务描述使用 **WYSIWYG 富文本编辑器**（原型用 Quill 1.3.7 snow 主题，生产可换），必须支持：

| 功能 | 必选 | 说明 |
|---|---|---|
| 加粗 / 斜体 / 下划线 / 删除线 | ✅ | 基础格式 |
| 1-3 级标题 | ✅ | 用于任务分段 |
| 有序 / 无序列表 | ✅ | 列验收点 / 需求点 |
| 引用块 | ✅ | 引用 PRD / 需求片段 |
| 行内代码 / 代码块 | ✅ | 贴示例代码 / 字段名 |
| 链接 | ✅ | 引用外部文档 |
| 表格 | ⏳ 二期 | 复杂数据展示 |
| 图片 | ⏳ 二期 | 贴 UI 稿等 |
| @ 提及人 | ⏳ 二期 | 拉人入任务 |

**数据格式**：以 **HTML 字符串** 存数据库（不存 Markdown）。前端渲染时**必须**走 XSS 过滤（推荐 DOMPurify）。

**长度限制**：建议 ≤ 50,000 字符（防滥用）。

### 5.2 流程 B：投标

#### 单人投标
- 填写：投标金额、交期、可选方案描述
- 同一任务同一主投标人最多只能有一条 `ACTIVE` 投标
- 撤回后可以重投，但必须保留历史投标记录，不覆盖旧投标

#### 多人协作投标
- 主对接人 A 填写：投标金额、交期、方案
- 在"协作成员"中**从人员接口下拉选择**其他成员
- 协作成员只需要被加入，不需要二次确认（MVP 简化）

> 数据口径：投标成员统一写入 `bid_member`。主投标人为 `OWNER`，协作者为 `COLLABORATOR`。`bid` 表不再单独保存 `bidder_id`，避免双重数据不一致。

### 5.3 流程 C：选标

1. 发布者浏览所有投标（**互相不可见**，A 不知道 B 的报价）
2. 选中一个投标
3. **可调整**：最终金额、最终交期、任务明细
4. 系统自动通知中标者，**其他投标者状态变为 LOST**（无主动通知）
5. 任务状态变 `ASSIGNED`，中标 `BidMember` 复制为 `TaskAssignee`

### 5.4 流程 D：直接指名

1. 发布者填写任务信息 + **被指名人**（从人员接口选）+ 金额
2. 状态直接变 `ASSIGNED`，**无需被指名人接受**（MVP 简化）
3. 被指名人收到通知

> 数据口径：直接指名不强行生成投标记录，任务直接写入 `primary_assignee_id`，并将实际参与人写入 `task_assignee`；招标选标才写 `assigned_bid_id`。

> 未来可加：被指名人拒绝 → 回到 OPEN。

### 5.5 流程 E：老板加塞 / 协商延期

> 老板有了新紧急任务，要求开发者放下手头的事先做新的。

1. 开发者接了新任务（`ASSIGNED`）
2. 开发者对**现有进行中的任务**发起"协商延期"
3. 线下与现有任务发布者沟通
4. 双方达成一致后，发布者点击"同意延期"
5. 系统创建 `request_type = CHANGE` 的 Change Request，`new_value.finalDelivery` 记录新交期
6. 确认通过后任务状态保持 `ASSIGNED`，生效结果写入 TaskEvent

> **不强制**：协商失败时，由双方自行找上司处理，平台不介入。

### 5.6 流程 F：任务变更 / 取消

#### 变更（进行中）
- 可变更字段：需求描述、最终金额、最终交期
- **必须双方确认**才生效；待确认阶段进入 Change Request
- 一个 CHANGE 请求可以同时修改上述一个或多个字段，必须整体同意或整体拒绝，MVP 不支持部分批准
- 延期只是修改 `finalDelivery` 的 CHANGE 请求，不设置单独延期类型
- 确认通过后的变更结果入 TaskEvent

#### 取消（任意状态）
- DRAFT / OPEN：发布者单方取消
- ASSIGNED：**必须双方同意**
- 进行中取消使用 `request_type = CANCEL` 的 Change Request；审批通过后 Task 状态变 `CANCELLED`、version 递增，并记录 `TASK_CANCELLED`
- 取消后状态 `CANCELLED`
- **MVP 不做已做工时的折算**（不记录工时）

### 5.7 流程 G：完成确认

1. 开发者提交交付记录（TaskDelivery），可附带交付附件
2. 发布者审核交付内容
3. 发布者可以验收通过，或要求修改
4. 某次交付被验收通过（`ACCEPTED`）后，任务才能变为 `COMPLETED`
5. 交付提交、验收、要求修改、完成任务都写入 TaskEvent

> 数据库支持一个任务多次交付（`delivery_no = 1, 2, 3...`），但 Phase 1 产品只开放一次交付：最多创建一条 TaskDelivery，固定 `delivery_no = 1`。发布者要求修改时，Task 状态保持 `ASSIGNED`，TaskDelivery 可进入 `REVISION_REQUIRED`，第一阶段通过线下沟通或更新现有交付信息处理，不创建第二条交付。

> MVP 不做超时自动机制。发布者不点完成 = 任务永远在 `ASSIGNED`，**双方自行找上司**处理。

### 5.8 流程 H：流标与重发

1. 未截止：`status = OPEN` 且 `bidding_deadline > 当前时间`，页面显示“招标中”，允许新投标、ACTIVE 投标撤回、发布者选标和取消。
2. 已截止且没有 ACTIVE 投标：定时任务将 Task 改为 `FAILED`，version 递增，写入 `TASK_FAILED`，页面显示“已流标”。
3. 已截止且存在 ACTIVE 投标：数据库状态仍保持 `OPEN`，页面派生显示“待选标”。普通投标人不能继续投标或撤回；发布者可以选标、延长招标截止时间、主动流标或取消任务。
4. “待选标”不是 TaskStatus 枚举值，筛选时通过查询条件实现，不新增 `PENDING_SELECTION` 或 `BIDDING_CLOSED`。
5. 发布者主动流标必须填写原因，Task 改为 `FAILED`，所有 ACTIVE Bid 改为 `LOST`，version 递增，并写入 `TASK_FAILED`。
6. 发布者可对 FAILED 任务点击"复制新建"，复制模板字段但不复制历史投标，新任务进入 `DRAFT`。

### 5.9 流程 I：任务详情页布局

任务详情页由上至下分 4 个区块，**描述必须做大**（不再像副标题）：

1. **任务标题 + 关键元信息卡**（顶部）
   - 标题：30px 加粗
   - 右侧：状态徽章
   - 下方一行：发布者头像 + 预算 + 期望交期 + 创建时间
2. **需求描述卡**（中部，重点）
   - 顶部：灰色小字标头 "需求描述"
   - 描述正文：16px 字号 / 1.75 行高 / 富文本渲染
   - 是页面的视觉中心
3. **补充信息卡**（中下）
   - 4 列网格：预算 / 最终金额 / 最终交期 / 招标截止
   - 付款账号单独占 2 列
4. **附件 / 投标列表 / 变更记录 / 操作按钮**（底部，按需显示）

> **设计意图**：让开发者**第一眼**就看清需求，而不是看一堆元信息。这是 v0.2 → v0.3 的关键改动之一。

### 5.10 流程 J：任务搜索

#### 5.10.1 适用范围
| 视图 | 支持搜索 | 搜索字段 |
|---|---|---|
| 任务大厅（/） | ✅ | 标题 + 描述 |
| 老板视图（/boss） | ✅ | 标题 |
| 个人中心 | ⏳ 二期 | — |

#### 5.10.2 交互规格
- **位置**：筛选条右上方，搜索框宽度 220-240px
- **触发方式**：实时筛选（onChange，**不**用回车键）
- **匹配规则**：
  - 大小写不敏感
  - 子串匹配（LIKE '%query%'）
  - MVP 不做分词、不做模糊匹配
- **清空**：搜索框右侧有 ✕ 按钮，一键清空
- **结果**：
  - 命中：列表立即刷新
  - 零命中：显示 "没找到匹配 'XXX' 的任务" + 清空按钮
- **与状态筛选的叠加**：搜索 ∩ 筛选（AND 关系）
- **重置分页**：任何搜索变更重置到第 1 页

#### 5.10.3 性能与限制
- MVP 数据量在 1k 以内，**MySQL LIKE 即可**（不强求 ES）
- 超过 10k 任务或需要模糊匹配时，再考虑 Elasticsearch / Meilisearch
- 搜索关键词不存历史记录（无搜索建议）

### 5.11 流程 K：列表分页

#### 5.11.1 适用范围
| 视图 | 分页 | 页大小 |
|---|---|---|
| 任务大厅（/） | ✅ | 4 / 页（演示用，生产可调到 10-20） |
| 老板视图 | ⏳ 二期 | — |
| 个人中心 | ⏳ 二期 | — |

> **老板视图和个人中心**目前任务数小（MVP 数据量 < 10），用筛选代替分页。

#### 5.11.2 交互规格
- **位置**：列表底部，仅在总条数 > 页大小时显示
- **组件**（Linear 风格）：
  - 左侧："显示 1-4 / 共 8 条"
  - 右侧："‹ 上一页" + "第 N / M 页" + "下一页 ›"
- **行为**：
  - 第 1 页时"上一页"禁用（变灰）
  - 末页时"下一页"禁用
  - **筛选 / 搜索 变更重置到第 1 页**
  - 直接跳转到指定页（MVP 不做，超出页大小按末页处理）

#### 5.11.3 后端实现
- API：`GET /api/tasks?page=1&size=4&q=xxx&status=OPEN`
- 返回结构：
  ```json
  {
    "items": [...],
    "total": 8,
    "page": 1,
    "size": 4,
    "totalPages": 2
  }
  ```

| uploaded_by | string | 人员工号（外部人员接口标识） |
| created_at | timestamp | 创建时间 |

### 6.7 TaskChangeRequest（任务变更请求）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 主键 |
| task_id | string | 关联任务 |
| request_no | int | 任务内变更请求编号 |
| request_type | enum | CHANGE/CANCEL |
| old_value | json | 变更前值，camelCase JSON |
| new_value | json | 变更后值，camelCase JSON |
| base_task_version | long | 发起变更时的 Task version |
| initiator_id | string | 人员工号（外部人员接口标识） |
| approver_id | string | 人员工号（外部人员接口标识） |
| status | enum | PENDING/APPROVED/REJECTED/CANCELLED |
| expires_at | timestamp | 过期时间，可为空 |
| responded_at | timestamp | 确认 / 拒绝时间 |
| cancelled_at | timestamp | 取消时间 |
| response_comment | text | 响应备注 |
| created_at | timestamp | 创建时间 |
| updated_at | timestamp | 更新时间 |

`old_value` 和 `new_value` 规范：

- `CHANGE` 只允许包含本次修改的 `finalAmount`、`finalDelivery`、`description` 字段；未修改字段不要写入 JSON。
- `CANCEL` 只允许把 `status` 从当前状态改为 `CANCELLED`，取消原因写入 `reason`。
- 字段名统一 camelCase；审批通过时必须校验 `task.version == base_task_version`。

```json
{
  "finalAmount": 3000,
  "finalDelivery": "2026-07-20"
}
```

```json
{
  "finalAmount": 3500,
  "finalDelivery": "2026-07-25"
}
```

### 6.8 TaskEvent（任务事件）

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 主键 |
| task_id | string | 关联任务 |
| event_type | enum | 任务事件类型 |
| operator_id | string | 人员工号（外部人员接口标识），系统事件可为空 |
| from_status | enum | 变更前任务状态 |
| to_status | enum | 变更后任务状态 |
| related_type | string | 关联对象类型 |
| related_id | string | 关联对象 ID |
| event_data | json | 事件快照 |
| remark | text | 备注 |
| created_at | timestamp | 创建时间 |

TaskEvent 只记录关键业务事件，不记录查看页面、搜索、下载附件、普通查询、草稿内容编辑或打开任务详情。事件类型为：`TASK_CREATED`、`TASK_PUBLISHED`、`BID_SUBMITTED`、`BID_WITHDRAWN`、`BID_ACCEPTED`、`TASK_ASSIGNED`、`CHANGE_REQUESTED`、`CHANGE_APPROVED`、`CHANGE_REJECTED`、`DELIVERY_SUBMITTED`、`DELIVERY_ACCEPTED`、`REVISION_REQUIRED`、`TASK_COMPLETED`、`TASK_CANCELLED`、`TASK_FAILED`、`BIDDING_DEADLINE_EXTENDED`。TaskEvent 不是 Event Sourcing、消息队列、事件总线、Outbox 或状态重建机制。

### 6.9 视图层（不存表，按需聚合）

- **老板甘特图视图**：所有任务 + 按时间线 + 按状态着色

## 9. 老板视图（甘特图）

### 9.1 入口
- 老板账号登录后，主导航有"老板视图"入口
- 普通用户**不可见**此入口

### 9.2 视图规格
- **图表类型**：甘特图（Gantt）
- **X 轴**：时间
- **Y 轴**：任务列表（默认按创建时间倒序）
- **任务条**：从任务创建到最终交期的时间跨度
- **颜色按状态**：
  - OPEN：蓝色（招标中）
  - ASSIGNED：橙色（进行中）
  - COMPLETED：绿色（已完成）
  - FAILED：灰色（流标）
  - CANCELLED：浅灰 + 删除线（已取消）

### 9.3 交互
- 点击任务条 → 跳转到任务详情
- 顶部筛选：按部门、按发布者、按状态
- 时间范围：默认最近 90 天，可选本周 / 本月 / 本季

### 9.4 数据范围
- 默认看**全部任务**
- 可按付款账号（部门）筛选
