## 1. 设计原则

| 原则 | 说明 |
|---|---|
| **按业务域拆，不按层拆** | 同一业务的 controller / service / mapper / entity 放一起，新人易上手 |
| **MVP 不做微服务** | 单体应用，方便部署和调试；二期再拆 |
| **数据库读写同库** | 不引入读写分离，QPS 不到需要的时候 |
| **不预判二期** | 不为"将来可能用"建表/字段 |

### 2.2 关键 Service 职责

| Service | 主要方法 | 事务边界 |
|---|---|---|
| `TaskService` | `createTask / directAssign / selectBid / completeTask / cancelTask / cloneTask / listTasks / getTaskDetail / updateExtension` | 单一任务操作，方法内事务 |
| `BidService` | `submitBid / withdrawBid / listBidsForTask` | 投标 / 撤回：单事务 |
| `AttachmentService` | `upload / download / listForTask / delete` | 上传：调 StorageService + 写记录，单事务 |
| `ChangeLogService` | `recordChange / listForTask` | 只读为主 |
| `UserService` | `getCurrentUser / getUserById` | 只读 + Redis 缓存 |

## 4. 数据库设计

### 4.1 表清单

| # | 表名 | 中文 | MVP 优先级 |
|---|---|---|---|
| 1 | `task` | 任务主表 | P0 |
| 2 | `bid` | 投标表 | P0 |
| 3 | `task_attachment` | 任务附件 | P0 |
| 4 | `task_change_log` | 任务变更记录 | P0 |
| — | `payment_account` | 付款账号 | ❌ 外部引用，不存 |
| — | `user_cache` | 用户缓存 | ❌ 用 Redis，不入 DB |
| — | `system_config` | 系统配置 | ❌ 二期 |

> **4 张核心表**足以支撑 MVP。其他表（信用、评分、推送历史）留二期。

### 4.3 ER 关系

```
┌─────────────┐
│    task     │
└──────┬──────┘
       │ 1
       │
       ├─────► bid (N)
       │         │
       │         │ collaborators (JSON, 不建关联表)
       │         ▼
       │      (user_id 引用外部系统)
       │
       ├─────► task_attachment (N)
       │
       └─────► task_change_log (N)
```

**关键点**：
- 不建立 user 表（人员信息在外部）
- 不建立 payment_account 表（外部引用）
- bid ↔ task：N:1，外键 `bid.task_id → task.id`
- 协作成员用 JSON 数组存 user_id（不建多对多表）

### 4.4 索引策略

**读多写少**的场景（任务大厅、个人主页、老板视图）都靠**复合索引**优化：

| 查询场景 | 索引 |
|---|---|
| 任务大厅：按状态 + 按时间倒序 | `(status, created_at DESC)` |
| 我的发布：按发布者 + 按状态 | `(created_by, status)` |
| 老板视图：按付款账号 + 状态 | `(payment_account_id, status)` |
| 投标列表：任务内 + 状态 | `(task_id, status)` |
| 我投标的：投标者 + 状态 | `(bidder_id, status)` |

**软删过滤**：所有表都有 `deleted_at`，索引中**不包含** `deleted_at`（避免索引膨胀）。SQL 中统一 `WHERE deleted_at IS NULL`。

## 5. 跨领域关注点

### 5.1 鉴权流程

```
HTTP Request
   │
   ▼
AuthInterceptor.preHandle
   │
   ├─ 提取 Authorization Header 的 token
   │
   ├─ 调 SsoClient.validate(token)
   │     └─ 失败 → 抛 401
   │
   ├─ 拿到 userId
   │
   ├─ 查 Redis 缓存 UserContext
   │     └─ 未命中 → 调 PersonnelClient + 写缓存
   │
   └─ UserContext 放入 RequestContext (ThreadLocal)
        └─ Controller 通过 @CurrentUser 注解拿
```

### 5.2 错误处理

- 全局 `@RestControllerAdvice` + `@ExceptionHandler`
- 业务异常用 `BusinessException` + `ErrorCode` 枚举
- 前端 axios 拦截器：401 → 跳登录；其他 → toast 错误信息

### 5.3 日志

- SLF4J + Logback
- 格式：`%d{yyyy-MM-dd HH:mm:ss} %-5level [%thread] %logger{36} - %msg%n`
- 关键操作记录 INFO（任务发布 / 选标 / 完成）
- 异常记录 ERROR（含 stacktrace）
- 开发期输出到控制台，生产期输出到文件

### 5.4 事务边界

- 默认 Spring `@Transactional`
- 关键操作显式标注（如 `selectBid` 要确保任务状态和 bid 状态同时变更）
- 不要在 Controller 调事务，事务在 Service 层

### 5.5 雪花 ID 生成

- 用 `hutool` 或 `baidu/fastjson` 的雪花 ID
- workerId 通过环境变量注入（避免多实例冲突）
- ID 可读性：`[41bit timestamp][10bit workerId][12bit sequence]`
