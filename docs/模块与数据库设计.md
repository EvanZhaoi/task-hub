## 1. 设计原则

| 原则 | 说明 |
|---|---|
| **按业务域拆，不按层拆** | 同一业务的 controller / service / mapper / entity 放一起，新人易上手 |
| **MVP 不做微服务** | 单体应用，方便部署和调试；二期再拆 |
| **数据库读写同库** | 不引入读写分离，QPS 不到需要的时候 |
| **不预判二期** | 不为"将来可能用"建表/字段 |

### 2.2 关键 Service 职责

| Service | 主要方法 | 事务边界 |
|---|---|---|
| `TaskService` | `createTask / directAssign / selectBid / completeTask / cancelTask / cloneTask / listTasks / getTaskDetail` | 单一任务操作，方法内事务 |
| `BidService` | `submitBid / withdrawBid / listBidsForTask` | 投标 / 撤回：单事务，写 BidMember + TaskEvent |
| `DeliveryService` | `submitDelivery / acceptDelivery / requestRevision / withdrawDelivery` | 交付状态变更 + TaskEvent；Phase 1 只允许创建一次交付 |
| `AttachmentRefService` | `attachToOwner / listForOwner / deleteRef` | 只记录外部附件 ID，不负责文件上传下载 |
| `ChangeRequestService` | `submitChange / approveChange / rejectChange / cancelChange / listPendingForTask` | `CHANGE` / `CANCEL` 审批通过时校验 task.version，更新任务 + 写 TaskEvent |
| `TaskEventService` | `recordEvent / listForTask` | 追加事件，支撑任务时间线 |
| `UserService` | `getCurrentUser / getUserById` | 只读 + Redis 缓存 |
| `BossService` | `listAllTasksForGantt` | 只读 |

## 4. 数据库设计

### 4.1 表清单

| # | 表名 | 中文 | MVP 优先级 |
|---|---|---|---|
| 1 | `task` | 任务主表 | P0 |
| 2 | `bid` | 投标表 | P0 |
| 3 | `bid_member` | 投标成员表 | P0 |
| 4 | `task_assignee` | 任务执行成员表 | P0 |
| 5 | `task_delivery` | 任务交付表 | P0 |
| 6 | `attachment_ref` | 附件引用表 | P0 |
| 7 | `task_change_request` | 任务变更请求表 | P0 |
| 8 | `task_event` | 任务事件表 | P0 |
| 9 | `taskhub_user_role` | TaskHub 用户角色表 | P0 |
| — | `payment_account` | 付款账号 | ❌ 外部引用，不存 |
| — | `user_cache` | 用户缓存 | ❌ 用 Redis，不入 DB |
| — | `notification` | 通知 | ❌ 二期 |

> **8 张核心业务表 + 1 张角色控制表**支撑 MVP。Task 保存当前状态，TaskEvent 保存历史事件；TaskChangeRequest 保存协商审批过程；AttachmentRef 只保存外部附件 ID；TaskHubUserRole 只保存工号与业务角色关系，不维护本地用户资料。

### 4.2 表结构详细

详细 DDL 见 [`database/schema.sql`](../database/schema.sql)。这里给设计说明。

#### 4.2.1 `task` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | 雪花 ID |
| `title` | VARCHAR(200) | ✓ | — | 任务标题 |
| `description` | LONGTEXT | — | FULLTEXT (二期) | 富文本 HTML |
| `payment_account_id` | VARCHAR(64) | ✓ | IDX | 付款账号 ID（外部引用） |
| `payment_account_snapshot` | JSON | — | — | 付款账号历史快照，仅用于展示和审计 |
| `budget` | DECIMAL(18,2) | ✓ | — | 预算金额 |
| `final_amount` | DECIMAL(18,2) | — | — | 最终成交金额 |
| `expected_delivery` | DATE | ✓ | — | 期望交期 |
| `final_delivery` | DATE | — | — | 最终交期 |
| `bidding_deadline` | TIMESTAMP | — | — | 招标截止时间（流标判定依据） |
| `status` | VARCHAR(20) | ✓ | IDX | DRAFT / OPEN / ASSIGNED / COMPLETED / FAILED / CANCELLED |
| `assignment_type` | VARCHAR(20) | ✓ | — | BIDDING / DIRECT |
| `complexity` | VARCHAR(20) | ✓ | IDX | LOW / MEDIUM / HIGH |
| `created_by` | VARCHAR(32) | ✓ | IDX | 人员工号（外部人员接口标识） |
| `created_by_snapshot` | JSON | — | — | 发布者历史快照，仅用于展示和审计 |
| `assigned_bid_id` | BIGINT UNSIGNED | — | — | 中标投标 ID（仅招标模式，冗余字段） |
| `primary_assignee_id` | VARCHAR(32) | — | IDX | 人员工号（外部人员接口标识） |
| `version` | BIGINT | ✓ | — | 乐观锁版本号 |
| `completed_at` | TIMESTAMP | — | — | 完成时间 |
| `created_at` | TIMESTAMP | ✓ | IDX | |
| `updated_at` | TIMESTAMP | ✓ | — | |
| `updated_by` | VARCHAR(32) | — | — | 人员工号（外部人员接口标识） |

**索引**：
- `PRIMARY (id)`
- `IDX_task_status_created (status, created_at DESC)` — 任务大厅列表
- `IDX_task_created_by_status_created (created_by, status, created_at DESC)` — 我发布的
- `IDX_task_primary_assignee_status (primary_assignee_id, status)` — 我负责的
- `IDX_task_payment_account_status (payment_account_id, status)` — 老板视图按付款账号筛选
- `IDX_task_bidding_deadline_status (bidding_deadline, status)` — 定时流标扫描
- `IDX_complexity_status (complexity, status)` — 任务大厅按复杂度筛选

**业务层约束**：
- BIDDING 模式：选标后 `assigned_bid_id` 和 `primary_assignee_id` 均非空。
- DIRECT 模式：`assigned_bid_id` 为空，`primary_assignee_id` 非空。
- BIDDING 模式下，`primary_assignee_id` 必须与中标 Bid 的 OWNER 工号一致。
- 权限判断必须使用实时外部人员接口，不能使用 JSON 快照。
- TaskHub 不维护本地用户表，所有人员引用字段均保存公司工号。

#### 4.2.2 `bid` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `task_id` | BIGINT UNSIGNED | ✓ | FK / IDX | 任务 ID |
| `amount` | DECIMAL(18,2) | ✓ | — | 投标金额 |
| `delivery_date` | DATE | ✓ | — | 承诺交付日期 |
| `proposal` | TEXT | — | — | 投标方案说明 |
| `status` | VARCHAR(20) | ✓ | IDX | ACTIVE / WITHDRAWN / ACCEPTED / LOST |
| `revision_no` | INT | ✓ | — | 修订序号，撤回后重投递增 |
| `active_key` | VARCHAR(160) | — | UNIQ | ACTIVE 时唯一标识 task_id + OWNER 工号 |
| `withdrawn_at` | TIMESTAMP | — | — | 撤回时间 |
| `created_at` | TIMESTAMP | ✓ | — | |
| `updated_at` | TIMESTAMP | ✓ | — | |

**设计说明**：Bid 不再单独保存 `bidder_id`，主投标人统一通过 `bid_member.role = OWNER` 表示，避免双重数据不一致。`active_key` 由业务层使用 `task_id` 和 OWNER 工号生成，例如 `10001:E12345`。`ACTIVE` 状态时非空，`WITHDRAWN` / `ACCEPTED` / `LOST` 时置为 `NULL`，唯一索引用于保证同一任务同一 OWNER 最多一条 ACTIVE 投标，同时允许保存撤回历史。提交 Bid、插入 OWNER / COLLABORATOR BidMember、写入 TaskEvent 必须在同一个事务中完成。

- `UNIQ_task_assignee_task_user (task_id, user_id)` — 同一任务下同一执行成员只出现一次。
- 每个任务只能有一个 OWNER，由业务层保证。
- MVP 执行期间不允许新增、删除或更换成员。

#### 4.2.5 `task_delivery` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `task_id` | BIGINT UNSIGNED | ✓ | FK / IDX | 任务 ID |
| `delivery_no` | INT | ✓ | UNIQ | 任务内交付序号 |
| `submitted_by` | VARCHAR(32) | ✓ | IDX | 人员工号（外部人员接口标识） |
| `description` | TEXT | — | — | 交付说明 |
| `status` | VARCHAR(30) | ✓ | IDX | SUBMITTED / ACCEPTED / REVISION_REQUIRED / WITHDRAWN |
| `submitted_at` | TIMESTAMP | ✓ | — | 提交时间 |
| `reviewed_by` | VARCHAR(32) | — | — | 人员工号（外部人员接口标识） |
| `reviewed_at` | TIMESTAMP | — | — | 评审时间 |
| `review_comment` | VARCHAR(1000) | — | — | 评审意见 |
| `created_at` | TIMESTAMP | ✓ | — | |
| `updated_at` | TIMESTAMP | ✓ | — | |

**约束**：`UNIQ_task_delivery_no (task_id, delivery_no)`，数据库支持一个任务多次提交交付，任务只有某次交付 `ACCEPTED` 后才能 `COMPLETED`。

**Phase 1 产品规则**：第一阶段只允许创建一条交付记录，固定 `delivery_no = 1`；不增加 `unique(task_id)`，避免未来开放多次交付时修改数据库约束。发布者要求修改时，Task 状态继续保持 `ASSIGNED`，当前 TaskDelivery 可进入 `REVISION_REQUIRED`；第一阶段通过线下沟通或更新现有交付信息处理，不创建 `delivery_no = 2`。

#### 4.2.6 `attachment_ref` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `owner_type` | VARCHAR(30) | ✓ | IDX | TASK / BID / DELIVERY / CHANGE_REQUEST |
| `owner_id` | BIGINT UNSIGNED | ✓ | IDX | 所属对象 ID |
| `attachment_id` | VARCHAR(128) | ✓ | — | 外部附件 ID |
| `uploaded_by` | VARCHAR(32) | ✓ | — | 人员工号（外部人员接口标识） |
| `created_at` | TIMESTAMP | ✓ | — | |

**设计说明**：采用多态关联，不建立数据库外键。业务层必须检查 `owner_id` 对应对象真实存在。`attachment_id` 指向外部文件服务，不保存真实文件内容。

#### 4.2.7 `task_change_request` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `task_id` | BIGINT UNSIGNED | ✓ | FK / IDX | 任务 ID |
| `request_no` | INT | ✓ | UNIQ | 任务内变更请求编号 |
| `request_type` | VARCHAR(20) | ✓ | — | CHANGE / CANCEL |
| `old_value` | JSON | — | — | 变更前值，camelCase JSON |
| `new_value` | JSON | — | — | 变更后值，camelCase JSON |
| `reason` | VARCHAR(500) | — | — | 原因 |
| `base_task_version` | BIGINT | ✓ | — | 发起时的 task.version |
| `initiator_id` | VARCHAR(32) | ✓ | — | 人员工号（外部人员接口标识） |
| `approver_id` | VARCHAR(32) | ✓ | IDX | 人员工号（外部人员接口标识） |
| `status` | VARCHAR(20) | ✓ | IDX | PENDING / APPROVED / REJECTED / CANCELLED |
| `expires_at` | TIMESTAMP | — | — | 过期时间，MVP 暂不自动过期 |
| `responded_at` | TIMESTAMP | — | — | 响应时间 |
| `cancelled_at` | TIMESTAMP | — | — | 取消时间 |
| `response_comment` | VARCHAR(1000) | — | — | 响应备注 |
| `created_at` | TIMESTAMP | ✓ | — | |
| `updated_at` | TIMESTAMP | ✓ | — | |

**JSON 格式约定**：
- `request_type = CHANGE`：允许一次修改 `finalAmount`、`finalDelivery`、`description` 中的一个或多个字段；未修改字段不要写入 JSON；一个请求必须整体同意或整体拒绝，MVP 不支持部分批准。
- `request_type = CANCEL`：只能把任务状态改成 `CANCELLED`，不允许同时修改金额、交期或描述；取消原因写入 `reason`。
- `old_value` / `new_value` 必须使用 camelCase 字段名。示例：`{"finalAmount": 3000, "finalDelivery": "2026-07-20"}` → `{"finalAmount": 3500, "finalDelivery": "2026-07-25"}`；取消示例：`{"status": "ASSIGNED"}` → `{"status": "CANCELLED"}`。
- 审批通过时必须校验 Task 当前 `version` 仍等于 `base_task_version`。MVP 同一任务最多只允许存在一个 `PENDING` 变更请求，该限制由业务层保证，不使用触发器或条件唯一索引。

#### 4.2.8 `task_event` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `task_id` | BIGINT UNSIGNED | ✓ | FK / IDX | 任务 ID |
| `event_type` | VARCHAR(40) | ✓ | IDX | 任务事件类型 |
| `operator_id` | VARCHAR(32) | — | IDX | 人员工号（外部人员接口标识），系统事件可为空 |
| `from_status` | VARCHAR(20) | — | — | 变更前任务状态 |
| `to_status` | VARCHAR(20) | — | — | 变更后任务状态 |
| `related_type` | VARCHAR(30) | — | — | 关联对象类型 |
| `related_id` | BIGINT UNSIGNED | — | — | 关联对象 ID |
| `event_data` | JSON | — | — | 事件快照 |
| `remark` | VARCHAR(500) | — | — | 备注 |
| `created_at` | TIMESTAMP | ✓ | IDX | |

**设计说明**：TaskEvent 是任务详情时间线和业务审计记录，不是 Event Sourcing、消息队列、事件总线、Outbox 或状态重建机制；Task 当前状态仍保存在 `task.status` 中，不反向重建 Task 当前状态。只记录关键业务动作，不记录查看页面、搜索、下载附件、普通查询、修改尚未发布的草稿内容或打开任务详情。

**事件类型**：`TASK_CREATED`、`TASK_PUBLISHED`、`BID_SUBMITTED`、`BID_WITHDRAWN`、`BID_ACCEPTED`、`TASK_ASSIGNED`、`CHANGE_REQUESTED`、`CHANGE_APPROVED`、`CHANGE_REJECTED`、`DELIVERY_SUBMITTED`、`DELIVERY_ACCEPTED`、`REVISION_REQUIRED`、`TASK_COMPLETED`、`TASK_CANCELLED`、`TASK_FAILED`、`BIDDING_DEADLINE_EXTENDED`。发布者主动流标和无投标自动流标都使用 `TASK_FAILED`，通过 `event_data.failureType` 区分：`MANUAL` 或 `NO_BID_TIMEOUT`。

### 4.3 ER 关系

```
task 1 ── N bid
bid  1 ── N bid_member
task 1 ── N task_assignee
task 1 ── N task_delivery
task 1 ── N task_change_request
task 1 ── N task_event

attachment_ref 通过 owner_type + owner_id 关联 task / bid / task_delivery / task_change_request
```

**关键点**：
- 不建立 user 表，人员引用统一保存公司工号，权限判断使用实时外部人员接口
- 不建立 payment_account 表（外部引用）
- bid_member 统一表示投标 OWNER 和 COLLABORATOR，不在 bid 中重复存主投标人
- 直接指名不强行生成 bid，直接写 `task.primary_assignee_id`，并写入 `task_assignee`
- 附件引用使用多态关联，不建立数据库外键
- 附件不存文件元信息，只存外部 `attachment_id`
- 业务核心数据原则上不物理删除，通过状态和事件保留历史

### 4.4 索引策略

| 查询场景 | 索引 |

## 5. 业务规则与跨领域关注点

### 5.1 招标截止后的状态规则

| 场景 | 数据库条件 | 数据库状态 | 页面展示 | 允许操作 |
|---|---|---|---|---|
| 未截止 | `status = OPEN` 且 `bidding_deadline > 当前时间` | OPEN | 招标中 | 新投标、ACTIVE 投标撤回、发布者选标、发布者取消任务 |
| 已截止且无 ACTIVE 投标 | `status = OPEN` 且 `bidding_deadline <= 当前时间` 且不存在 ACTIVE Bid | 定时任务改为 FAILED | 已流标 | 发布者复制新建 |
| 已截止且存在 ACTIVE 投标 | `status = OPEN` 且 `bidding_deadline <= 当前时间` 且存在 ACTIVE Bid | 仍保持 OPEN | 待选标 | 发布者选标、延长截止时间、主动流标、取消任务 |

规则说明：
- `待选标` 是查询和页面派生展示状态，不是 `TaskStatus` 枚举值；筛选待选标时使用上述查询条件实现，不新增 `PENDING_SELECTION` 或 `BIDDING_CLOSED`。
- 截止后普通投标人不能继续投标，不能撤回已有 ACTIVE 投标，但可以查看自己的投标。
- 延长招标截止时间要求任务仍为 `OPEN`，新截止时间必须是未来时间，任务进入 `ASSIGNED` 后不能再延长；延长后重新允许投标和撤回，并写入 `BIDDING_DEADLINE_EXTENDED` TaskEvent。
- 发布者主动流标必须填写原因，Task 改为 `FAILED`，所有 ACTIVE Bid 改为 `LOST` 且 `active_key` 置空，Task version 递增，写入 `TASK_FAILED`。`event_data` 示例：`{"failureType":"MANUAL","reason":"现有方案不符合需求","bidCount":3}`。
- 无投标自动流标写入 `TASK_FAILED`，`event_data` 示例：`{"failureType":"NO_BID_TIMEOUT"}`。

### 5.2 鉴权流程

```
HTTP Request
   │
   ▼
AuthInterceptor.preHandle
   │
   ├─ 提取 Authorization Header 的 token
   │
   ├─ 调 SsoClient.validate(token)
   │     └─ 失败 → 抛 401
   │
   ├─ 拿到人员工号
   │
   ├─ 查 Redis 缓存 UserContext
   │     └─ 未命中 → 调 PersonnelClient + 写缓存
   │
   └─ UserContext 放入 RequestContext (ThreadLocal)
        └─ Controller 通过 @CurrentUser 注解拿
```

### 5.3 错误处理

- 全局 `@RestControllerAdvice` + `@ExceptionHandler`
- 业务异常用 `BusinessException` + `ErrorCode` 枚举
- 前端 axios 拦截器：401 → 跳登录；其他 → toast 错误信息

### 5.4 日志

- SLF4J + Logback
- 格式：`%d{yyyy-MM-dd HH:mm:ss} %-5level [%thread] %logger{36} - %msg%n`
- 关键操作记录 INFO（任务发布 / 选标 / 完成）
- 异常记录 ERROR（含 stacktrace）
- 开发期输出到控制台，生产期输出到文件

### 5.5 事务边界

- 默认 Spring `@Transactional`
- 关键操作显式标注（如 `selectBid` 要确保任务状态和 bid 状态同时变更）
- 不要在 Controller 调事务，事务在 Service 层

**投标事务**：

1. 校验 Task 为 `OPEN`。
2. 校验当前时间未超过 `bidding_deadline`。
3. 校验 OWNER 没有其他 `ACTIVE` 投标。
4. 插入 Bid，并生成 `active_key = task_id + ':' + OWNER 工号`。
5. 插入 OWNER BidMember。
6. 插入 COLLABORATOR BidMember。
7. 写入 `BID_SUBMITTED` TaskEvent。
8. 提交事务。

**撤回投标事务**：

1. 校验当前时间未超过 `bidding_deadline`，截止后不允许撤回。
2. 校验 Bid 为 `ACTIVE`。
3. 校验操作者为 OWNER。
4. Bid 改为 `WITHDRAWN`。
5. `active_key` 置 `NULL`。
6. 写入 `BID_WITHDRAWN` TaskEvent。
7. 提交事务。

**选标事务**：

1. 查询并锁定 Task。
2. 校验 Task 状态仍为 `OPEN`。
3. 校验目标 Bid 状态仍为 `ACTIVE`。
4. 目标 Bid 改为 `ACCEPTED`，`active_key` 置 `NULL`。
5. 同一任务下其他 `ACTIVE` Bid 改为 `LOST`，`active_key` 置 `NULL`。
6. Task 改为 `ASSIGNED`。
7. 写入 `assigned_bid_id`。
8. 设置 `primary_assignee_id` 为 BidMember OWNER 的工号。
9. 将中标 `BidMember` 复制到 `TaskAssignee`，OWNER 复制为 OWNER，COLLABORATOR 复制为 COLLABORATOR。
10. Task version 递增。
11. 写入 `BID_ACCEPTED` 和 `TASK_ASSIGNED` TaskEvent。
12. 提交事务。

**变更审批事务**：

1. 锁定 Task 和 ChangeRequest。
2. 校验 ChangeRequest 为 `PENDING`。
3. 校验 `task.version` 与 `base_task_version` 一致。
4. 根据 `request_type` 应用 `CHANGE` 或 `CANCEL`。
5. Task version 递增。
6. ChangeRequest 改为 `APPROVED`。
7. 写入 `CHANGE_APPROVED` 或 `TASK_CANCELLED` TaskEvent。
8. 提交事务。

**无投标自动流标事务**：

1. 查找 `OPEN` 且已截止的任务。
2. 锁定目标 Task。
3. 再次确认不存在 `ACTIVE` Bid。
4. Task 改为 `FAILED`。
5. Task version 递增。
6. 写入 `TASK_FAILED` TaskEvent。
7. 提交事务。

上述流程必须在一个数据库事务内完成；MVP 不使用分布式事务、事件总线或数据库触发器。

### 5.6 软删除与可见性

MVP 不增加 `deleted`、`deleted_at`、`deleted_by` 或通用逻辑删除框架。TaskHub 目标是公开保留所有任务，已完成、已取消、已流标和失败任务都需要正常查询；投标、交付、变更和事件都属于业务历史。

如果未来出现合规隐藏、数据纠错或管理员撤下需求，可在独立版本中增加逻辑删除或可见性控制；优先考虑 `visibility_status` 或独立归档 / 隐藏能力，而不是直接删除业务历史。

### 5.7 雪花 ID 生成

- 当前 TaskHub 是单体 Laravel 项目，短期内不做多实例部署，因此不引入 WorkerId。
- 应用内使用简化版 Snowflake：`currentMillisecond * 1000 + sequence`。
- `sequence` 是同一毫秒内自增序号，范围 `0~999`；同一毫秒超过 1000 个 ID 时等待下一毫秒。
- 如果未来进入多实例或分布式部署，再升级为带 WorkerId 的完整雪花 ID，或接入公司统一 ID 服务。
