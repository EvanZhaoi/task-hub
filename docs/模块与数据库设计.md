## 1. 设计原则

| 原则 | 说明 |
|---|---|
| **按业务域拆，不按层拆** | 同一业务的 controller / service / mapper / entity 放一起，新人易上手 |
| **MVP 不做微服务** | 单体应用，方便部署和调试；二期再拆 |
| **数据库读写同库** | 不引入读写分离，QPS 不到需要的时候 |
| **不预判二期** | 不为"将来可能用"建表/字段 |

### 2.2 关键 Service 职责

| Service | 主要方法 | 事务边界 |
|---|---|---|
| `TaskService` | `createTask / directAssign / selectBid / completeTask / cancelTask / cloneTask / listTasks / getTaskDetail` | 单一任务操作，方法内事务 |
| `BidService` | `submitBid / withdrawBid / listBidsForTask` | 投标 / 撤回：单事务 |
| `AttachmentRefService` | `attachToTask / attachToBid / listForOwner / deleteRef` | 只记录外部附件 ID，不负责文件上传下载 |
| `ChangeRequestService` | `submitChange / approveChange / rejectChange / listPendingForTask` | 审批通过时同事务更新任务 + 写 Change Log |
| `ChangeLogService` | `recordChange / listForTask` | 只读为主 |
| `UserService` | `getCurrentUser / getUserById` | 只读 + Redis 缓存 |
| `BossService` | `listAllTasksForGantt` | 只读 |

## 4. 数据库设计

### 4.1 表清单

| # | 表名 | 中文 | MVP 优先级 |
|---|---|---|---|
| 1 | `task` | 任务主表 | P0 |
| 2 | `bid` | 投标表 | P0 |
| 3 | `task_assignee` | 任务最终参与人 | P0 |
| 4 | `bid_collaborator` | 投标协作人 | P0 |
| 5 | `attachment_ref` | 附件引用 | P0 |
| 6 | `task_change_request` | 任务变更请求 / 双方确认 | P0 |
| 7 | `task_change_log` | 任务变更记录 | P0 |
| — | `payment_account` | 付款账号 | ❌ 外部引用，不存 |
| — | `user_cache` | 用户缓存 | ❌ 用 Redis，不入 DB |
| — | `system_config` | 系统配置 | ❌ 二期 |

> **7 张核心表**支撑 MVP。`attachment_ref` 只保存外部附件 ID；`task_change_request` 承载待确认流程，`task_change_log` 只记录已经生效的结果。其他表（信用、评分、推送历史）留二期。

### 4.2 表结构详细

详细 DDL 见 [`database/schema.sql`](../database/schema.sql)。这里给设计说明。

**索引**：
- `PRIMARY (id)`

- `IDX_deleted (deleted_at)` — 软删过滤

| `deleted_at` | TIMESTAMP | — | IDX | |
| `pending_change_type` | TINYINT | — | UNIQ | 生成列：仅 PENDING 且未删除时等于 `change_type` |

**索引**：
- `PRIMARY (id)`

### 4.3 ER 关系

```
task 1 ── N bid
task 1 ── N task_assignee
bid  1 ── N bid_collaborator
task 1 ── N task_change_request
task 1 ── N task_change_log

attachment_ref 通过 owner_type + owner_id 关联 task 或 bid
```

**关键点**：
- 不建立 user 表（人员信息在外部）
- 不建立 payment_account 表（外部引用）
- bid ↔ task：N:1，外键 `bid.task_id → task.id`
- 直接指名不强行生成 bid，直接写 `task.assigned_user_id`，并写入 `task_assignee`
- 多人投标用 `bid_collaborator` 关系表，不用 JSON
- 附件不存文件元信息，只存外部 `attachment_id`

### 4.4 索引策略

**读多写少**的场景（任务大厅、个人主页、老板视图）都靠**复合索引**优化：

| 查询场景 | 索引 |
|---|---|
| 任务大厅：按状态 + 按时间倒序 | `(status, created_at DESC)` |
| 我的发布：按发布者 + 按状态 | `(created_by, status)` |
| 我接的：按负责人 + 按状态 | `(assigned_user_id, status)` |
| 我参与的任务：按参与人 | `task_assignee(user_id, role)` |
| 老板视图：按付款账号 + 状态 | `(payment_account_id, status)` |
| 任务大厅：按复杂度 + 按状态 | `(complexity, status)` |
| 投标列表：任务内 + 状态 | `(task_id, status)` |
| 我投标的：投标者 + 状态 | `(bidder_id, status)` |
| 我协作的投标：协作人 | `bid_collaborator(user_id)` |
| 附件列表：业务对象 | `attachment_ref(owner_type, owner_id)` |
| 我待确认的变更：确认人 + 状态 | `(approver_id, status)` |

**软删过滤**：业务表使用 `deleted_at`。涉及“未删除唯一性”的地方用生成列实现唯一约束，避免 MySQL 对 `NULL` 唯一索引的特殊行为导致约束失效。

## 5. 跨领域关注点

### 5.1 鉴权流程

```
HTTP Request
   │
   ▼
AuthInterceptor.preHandle
   │
   ├─ 提取 Authorization Header 的 token
   │
   ├─ 调 SsoClient.validate(token)
   │     └─ 失败 → 抛 401
   │
   ├─ 拿到 userId
   │
   ├─ 查 Redis 缓存 UserContext
   │     └─ 未命中 → 调 PersonnelClient + 写缓存
   │
   └─ UserContext 放入 RequestContext (ThreadLocal)
        └─ Controller 通过 @CurrentUser 注解拿
```

### 5.2 错误处理

- 全局 `@RestControllerAdvice` + `@ExceptionHandler`
- 业务异常用 `BusinessException` + `ErrorCode` 枚举
- 前端 axios 拦截器：401 → 跳登录；其他 → toast 错误信息

### 5.3 日志

- SLF4J + Logback
- 格式：`%d{yyyy-MM-dd HH:mm:ss} %-5level [%thread] %logger{36} - %msg%n`
- 关键操作记录 INFO（任务发布 / 选标 / 完成）
- 异常记录 ERROR（含 stacktrace）
- 开发期输出到控制台，生产期输出到文件

### 5.4 事务边界

- 默认 Spring `@Transactional`
- 关键操作显式标注（如 `selectBid` 要确保任务状态和 bid 状态同时变更）
- 不要在 Controller 调事务，事务在 Service 层

### 5.5 雪花 ID 生成

- 用 `hutool` 或 `baidu/fastjson` 的雪花 ID
- workerId 通过环境变量注入（避免多实例冲突）
- ID 可读性：`[41bit timestamp][10bit workerId][12bit sequence]`
