## 1. 设计原则

| 原则 | 说明 |
|---|---|
| **按业务域拆，不按层拆** | 同一业务的 controller / service / mapper / entity 放一起，新人易上手 |
| **MVP 不做微服务** | 单体应用，方便部署和调试；二期再拆 |
| **数据库读写同库** | 不引入读写分离，QPS 不到需要的时候 |
| **不预判二期** | 不为"将来可能用"建表/字段 |

### 2.2 关键 Service 职责

| Service | 主要方法 | 事务边界 |
|---|---|---|
| `TaskService` | `createTask / directAssign / selectBid / completeTask / cancelTask / cloneTask / listTasks / getTaskDetail` | 单一任务操作，方法内事务 |
| `BidService` | `submitBid / withdrawBid / listBidsForTask` | 投标 / 撤回：单事务，写 BidMember + TaskEvent |
| `DeliveryService` | `submitDelivery / acceptDelivery / requestRevision / withdrawDelivery` | 交付状态变更 + TaskEvent |
| `AttachmentRefService` | `attachToOwner / listForOwner / deleteRef` | 只记录外部附件 ID，不负责文件上传下载 |
| `ChangeRequestService` | `submitChange / approveChange / rejectChange / cancelChange / listPendingForTask` | 审批通过时校验 task.version，更新任务 + 写 TaskEvent |
| `TaskEventService` | `recordEvent / listForTask` | 追加事件，支撑任务时间线 |
| `UserService` | `getCurrentUser / getUserById` | 只读 + Redis 缓存 |
| `BossService` | `listAllTasksForGantt` | 只读 |

## 4. 数据库设计

### 4.1 表清单

| # | 表名 | 中文 | MVP 优先级 |
|---|---|---|---|
| 1 | `task` | 任务主表 | P0 |
| 2 | `bid` | 投标表 | P0 |
| 3 | `bid_member` | 投标成员表 | P0 |
| 4 | `task_assignee` | 任务执行成员表 | P0 |
| 5 | `task_delivery` | 任务交付表 | P0 |
| 6 | `attachment_ref` | 附件引用表 | P0 |
| 7 | `task_change_request` | 任务变更请求表 | P0 |
| 8 | `task_event` | 任务事件表 | P0 |
| — | `payment_account` | 付款账号 | ❌ 外部引用，不存 |
| — | `user_cache` | 用户缓存 | ❌ 用 Redis，不入 DB |
| — | `notification` | 通知 | ❌ 二期 |

> **8 张核心表**支撑 MVP。Task 保存当前状态，TaskEvent 保存历史事件；TaskChangeRequest 保存协商审批过程；AttachmentRef 只保存外部附件 ID。

### 4.2 表结构详细

详细 DDL 见 [`database/schema.sql`](../database/schema.sql)。这里给设计说明。

#### 4.2.1 `task` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | 雪花 ID |
| `title` | VARCHAR(200) | ✓ | — | 任务标题 |
| `description` | LONGTEXT | — | FULLTEXT (二期) | 富文本 HTML |
| `payment_account_id` | VARCHAR(64) | ✓ | IDX | 付款账号 ID（外部引用） |
| `payment_account_snapshot` | JSON | — | — | 付款账号历史快照，仅用于展示和审计 |
| `budget` | DECIMAL(18,2) | ✓ | — | 预算金额 |
| `final_amount` | DECIMAL(18,2) | — | — | 最终成交金额 |
| `expected_delivery` | DATE | ✓ | — | 期望交期 |
| `final_delivery` | DATE | — | — | 最终交期 |
| `bidding_deadline` | TIMESTAMP | — | — | 招标截止时间（流标判定依据） |
| `status` | VARCHAR(20) | ✓ | IDX | DRAFT / OPEN / ASSIGNED / COMPLETED / FAILED / CANCELLED |
| `assignment_type` | VARCHAR(20) | ✓ | — | BIDDING / DIRECT |
| `complexity` | VARCHAR(20) | ✓ | IDX | LOW / MEDIUM / HIGH |
| `created_by` | BIGINT UNSIGNED | ✓ | IDX | 发布者 user_id |
| `created_by_snapshot` | JSON | — | — | 发布者历史快照，仅用于展示和审计 |
| `assigned_bid_id` | BIGINT UNSIGNED | — | — | 中标投标 ID（仅招标模式，冗余字段） |
| `primary_assignee_id` | BIGINT UNSIGNED | — | IDX | 当前主负责人 user_id |
| `version` | BIGINT | ✓ | — | 乐观锁版本号 |
| `completed_at` | TIMESTAMP | — | — | 完成时间 |
| `created_at` | TIMESTAMP | ✓ | IDX | |
| `updated_at` | TIMESTAMP | ✓ | — | |
| `updated_by` | BIGINT UNSIGNED | — | — | |

**索引**：
- `PRIMARY (id)`
- `IDX_task_status_created (status, created_at DESC)` — 任务大厅列表
- `IDX_task_created_by_status_created (created_by, status, created_at DESC)` — 我发布的
- `IDX_task_primary_assignee_status (primary_assignee_id, status)` — 我负责的
- `IDX_task_payment_account_status (payment_account_id, status)` — 老板视图按付款账号筛选
- `IDX_task_bidding_deadline_status (bidding_deadline, status)` — 定时流标扫描
- `IDX_complexity_status (complexity, status)` — 任务大厅按复杂度筛选

**业务层约束**：
- BIDDING 模式：选标后 `assigned_bid_id` 和 `primary_assignee_id` 均非空。
- DIRECT 模式：`assigned_bid_id` 为空，`primary_assignee_id` 非空。
- BIDDING 模式下，`primary_assignee_id` 必须与中标 Bid 的 OWNER 成员一致。
- 权限判断必须使用实时外部人员接口，不能使用 JSON 快照。

#### 4.2.2 `bid` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `task_id` | BIGINT UNSIGNED | ✓ | FK / IDX | 任务 ID |
| `amount` | DECIMAL(18,2) | ✓ | — | 投标金额 |
| `delivery_date` | DATE | ✓ | — | 承诺交付日期 |
| `proposal` | TEXT | — | — | 投标方案说明 |
| `status` | VARCHAR(20) | ✓ | IDX | ACTIVE / WITHDRAWN / ACCEPTED / LOST |
| `revision_no` | INT | ✓ | — | 修订序号，撤回后重投递增 |
| `active_key` | VARCHAR(160) | — | UNIQ | ACTIVE 时唯一标识 task_id + OWNER user_id |
| `withdrawn_at` | TIMESTAMP | — | — | 撤回时间 |
| `created_at` | TIMESTAMP | ✓ | — | |
| `updated_at` | TIMESTAMP | ✓ | — | |

**设计说明**：Bid 不再单独保存 `bidder_id`，主投标人统一通过 `bid_member.role = OWNER` 表示，避免双重数据不一致。`active_key` 由业务层生成，保证同一任务同一 OWNER 最多一条 ACTIVE 投标，同时允许保存撤回历史。

#### 4.2.3 `bid_member` 表

| 字段 | 类型 | 必填 | 索引 | 说明 |
|---|---|---|---|---|
| `id` | BIGINT UNSIGNED | ✓ | PK | |
| `bid_id` | BIGINT UNSIGNED | ✓ | FK / IDX | 投标 ID |
| `user_id` | BIGINT UNSIGNED | ✓ | IDX | 成员 user_id |
| `role` | VARCHAR(20) | ✓ | — | OWNER / COLLABORATOR |
| `created_at` | TIMESTAMP | ✓ | — | |

**约束**：
- `UNIQ_bid_member_bid_user (bid_id, user_id)` — 同一 Bid 下同一成员只出现一次。
- 每个 Bid 必须且只能有一个 OWNER，由业务层保证。

### 4.3 ER 关系

```
task 1 ── N bid
bid  1 ── N bid_member
task 1 ── N task_assignee
task 1 ── N task_delivery
task 1 ── N task_change_request
task 1 ── N task_event

attachment_ref 通过 owner_type + owner_id 关联 task / bid / task_delivery / task_change_request
```

**关键点**：
- 不建立 user 表（人员信息在外部）
- 不建立 payment_account 表（外部引用）
- bid_member 统一表示投标 OWNER 和 COLLABORATOR，不在 bid 中重复存主投标人
- 直接指名不强行生成 bid，直接写 `task.primary_assignee_id`，并写入 `task_assignee`
- 附件引用使用多态关联，不建立数据库外键
- 附件不存文件元信息，只存外部 `attachment_id`
- 业务核心数据原则上不物理删除，通过状态和事件保留历史

### 4.4 索引策略

| 查询场景 | 索引 |
|---|---|
| 任务大厅：按状态 + 时间倒序 | `task(status, created_at)` |
| 我的发布 | `task(created_by, status, created_at)` |
| 我负责的 | `task(primary_assignee_id, status)` |
| 老板视图按付款账号 | `task(payment_account_id, status)` |
| 流标扫描 | `task(bidding_deadline, status)` |
| 复杂度筛选 | `task(complexity, status)` |
| 任务投标列表 | `bid(task_id, status)` |
| 我参与的投标 | `bid_member(user_id, bid_id)` |
| 我参与的任务 | `task_assignee(user_id, task_id)` |
| 附件列表 | `attachment_ref(owner_type, owner_id)` |
| 任务交付序号唯一 | `task_delivery(task_id, delivery_no)` |
| 变更请求列表 | `task_change_request(task_id, status, created_at)` |
| 我待确认的变更 | `task_change_request(approver_id, status, created_at)` |
| 任务时间线 | `task_event(task_id, created_at)` |

任务标题和描述的 `LIKE '%keyword%'` 搜索，MVP 暂不增加全文索引或 Elasticsearch；数据量较小时直接使用 LIKE。

## 5. 跨领域关注点

### 5.1 鉴权流程

```
HTTP Request
   │
   ▼
AuthInterceptor.preHandle
   │
   ├─ 提取 Authorization Header 的 token
   │
   ├─ 调 SsoClient.validate(token)
   │     └─ 失败 → 抛 401
   │
   ├─ 拿到 userId
   │
   ├─ 查 Redis 缓存 UserContext
   │     └─ 未命中 → 调 PersonnelClient + 写缓存
   │
   └─ UserContext 放入 RequestContext (ThreadLocal)
        └─ Controller 通过 @CurrentUser 注解拿
```

### 5.2 错误处理

- 全局 `@RestControllerAdvice` + `@ExceptionHandler`
- 业务异常用 `BusinessException` + `ErrorCode` 枚举
- 前端 axios 拦截器：401 → 跳登录；其他 → toast 错误信息

### 5.3 日志

- SLF4J + Logback
- 格式：`%d{yyyy-MM-dd HH:mm:ss} %-5level [%thread] %logger{36} - %msg%n`
- 关键操作记录 INFO（任务发布 / 选标 / 完成）
- 异常记录 ERROR（含 stacktrace）
- 开发期输出到控制台，生产期输出到文件

### 5.4 事务边界

- 默认 Spring `@Transactional`
- 关键操作显式标注（如 `selectBid` 要确保任务状态和 bid 状态同时变更）
- 不要在 Controller 调事务，事务在 Service 层

**选标事务要求**：

1. 查询并锁定 Task。
2. 校验 Task 状态仍为 `OPEN`。
3. 校验目标 Bid 状态仍为 `ACTIVE`。
4. 目标 Bid 改为 `ACCEPTED`。
5. 同一任务下其他 `ACTIVE` Bid 改为 `LOST`。
6. Task 改为 `ASSIGNED`。
7. 写入 `primary_assignee_id` 和 `assigned_bid_id`。
8. 将中标 `BidMember` 复制到 `TaskAssignee`。
9. 写入 `TaskEvent`。
10. 提交事务。

上述流程必须在一个数据库事务内完成；MVP 不使用分布式事务、事件总线或数据库触发器。

### 5.5 雪花 ID 生成

- 用 `hutool` 或 `baidu/fastjson` 的雪花 ID
- workerId 通过环境变量注入（避免多实例冲突）
- ID 可读性：`[41bit timestamp][10bit workerId][12bit sequence]`
