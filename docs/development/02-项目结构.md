# 02-项目结构

## 1. 本章节目标

本章解释 TaskHub Phase 1 的目录结构。目标是保持 Laravel 官方目录，不创建 DDD、Application、Infrastructure、Repository 等复杂分层。

## 2. 涉及文件

| 目录 | 作用 |
|---|---|
| `app/Models` | Eloquent Model，对应数据库表 |
| `app/Auth` | Laravel Auth 框架适配，不保存本地用户 |
| `app/Services` | 轻量业务服务，例如当前用户服务 |
| `app/Integrations/Sso` | 外部 SSO 集成代码 |
| `app/Http/Middleware` | HTTP 中间件，例如 Inertia 请求处理 |
| `config` | 应用配置 |
| `routes/web.php` | Web 路由，Inertia 页面入口 |
| `resources/js` | React + TypeScript 前端入口 |
| `resources/views/app.blade.php` | Inertia 挂载壳 |
| `tests` | Pest 测试 |

## 3. 实现步骤

1. 保留 Laravel 官方 `app/config/routes/resources/tests` 目录。
2. 新增 `app/Integrations/Sso` 放公司 SSO 对接。
3. 新增 `app/Services` 放统一 CurrentUserService。
4. 在 `resources/js` 下建立 React/Inertia 入口。
5. 不创建 Repository、Domain、Application、Infrastructure。

## 4. 核心代码讲解

Laravel 的 `routes/web.php` 类似 Spring MVC Controller 的路由注册，但可以直接写闭包。后续复杂业务再创建 Controller。

`app/Models` 的 Eloquent Model 类似 JPA Entity + Active Record。它既描述表结构映射，也提供查询入口，因此当前不再包一层 Repository。

`app/Services` 类似 Spring 的 `@Service`，但 Laravel 不需要注解；只要构造函数类型明确，容器就能自动注入。

## 5. 设计原因

当前阶段只做基础架构，目录越简单越容易接手。

没有采用复杂分层的原因：

- DDD/CQRS/Event Sourcing 对当前 MVP 没有直接价值。
- Repository 会把 Eloquent 的能力重复包装一遍。
- 微服务会增加部署、认证和事务复杂度。

## 6. 注意事项

- Service 可以有，但必须是明确职责，不要把所有代码塞进一个通用 Service。
- Model 不写业务流程，只写映射和关系。
- Controller 后续只负责编排请求和响应，不直接读取 Session 或写 SSO 逻辑。

## 7. 本阶段完成效果

团队成员看到目录后应能判断：

- 数据映射在 `app/Models`。
- 当前登录用户从 `app/Services/CurrentUserService.php` 获取。
- Laravel Auth 内部占位适配在 `app/Auth/SsoUserProvider.php`，不是本地用户体系。
- 外部 SSO 调用在 `app/Integrations/Sso`。
- React 页面从 `resources/js/Pages` 开始。
