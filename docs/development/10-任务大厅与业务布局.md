# 10-任务大厅与业务布局

## 本章目标

本章开始进入正式业务开发。目标不是做一个静态页面，而是把 TaskHub 的“任务大厅”做成后续业务功能的承载页面：

- React 页面有统一业务布局。
- Laravel Controller 从 MySQL 查询真实 `task` 数据。
- 页面支持关键词、状态、复杂度筛选。
- 页面支持分页。
- “待选标”作为派生状态展示，不新增数据库状态。
- SSO 退出继续使用原生 POST Form，避免外部 SSO 跳转触发 CORS。

## 学习目标

读完本章你应该理解：

- 为什么业务系统需要先抽 `AppLayout`。
- Laravel Controller 如何读取 Query String。
- Eloquent 如何组合 `where`、`whereHas`、`withCount` 和 `paginate`。
- Inertia 如何把分页数据传给 React。
- React 为什么用 GET 表单做筛选。
- 为什么“待选标”是展示状态，不是 `task.status`。
- Feature Test 如何为页面查询准备最小数据表和样例数据。

## 第一步：升级 TaskController

文件：

```text
app/Http/Controllers/TaskController.php
```

核心思路：

```php
public function index(Request $request): Response
{
    // 从 URL Query String 读取筛选条件，例如 /tasks?status=OPEN。
    $filters = $this->filters($request);

    // 先构造 Eloquent 查询，不立刻执行。
    $query = Task::query()
        ->select([...])
        ->withCount([
            // 查询每个任务当前 ACTIVE 投标数量，用于展示和派生“待选标”。
            'bids as active_bid_count' => fn (Builder $query): Builder => $query->where('status', 'ACTIVE'),
        ]);

    // 根据筛选条件继续拼接 SQL。
    $this->applyFilters($query, $filters);

    // paginate 执行查询并返回分页结果。
    $paginator = $query
        ->latest('created_at')
        ->paginate(10)
        ->withQueryString();

    return Inertia::render('Tasks/Index', [
        'filters' => $filters,
        'tasks' => [...],
    ]);
}
```

这里和 Spring Boot 的关系：

| Laravel | Spring Boot |
|---|---|
| `Request $request` | `HttpServletRequest` 或 Controller 方法参数 |
| `Task::query()` | JPA Repository / QueryDSL 查询入口 |
| `where()` | SQL where 条件 |
| `whereHas()` | 根据关联表存在性筛选 |
| `withCount()` | 子查询统计关联数量 |
| `paginate()` | Page / Pageable |
| `Inertia::render()` | 返回页面模型和视图 |

为什么不用 Repository：

当前列表查询只服务一个页面，直接放在 Controller 私有方法里更直观。等查询被多个 Controller 复用，或者出现复杂权限组合时，再抽 Service 会更合适。

## 第二步：理解“待选标”派生状态

数据库中 `task.status` 只有：

```text
DRAFT / OPEN / ASSIGNED / COMPLETED / FAILED / CANCELLED
```

没有：

```text
PENDING_SELECTION
```

“待选标”的规则是：

```text
task.status = OPEN
AND bidding_deadline <= 当前时间
AND 存在 ACTIVE 投标
```

所以 Controller 中这样处理：

```php
if ($filters['status'] === 'PENDING_SELECTION') {
    $query
        ->where('status', 'OPEN')
        ->where('bidding_deadline', '<=', now())
        ->whereHas('bids', fn (Builder $query): Builder => $query->where('status', 'ACTIVE'));

    return;
}
```

这样做的原因：

- 不增加数据库状态，避免状态机变复杂。
- 保留 `OPEN`，发布者仍可以选标、延长截止或主动流标。
- 页面筛选“待选标”时用查询条件实现。

## 第三步：抽 AppLayout

文件：

```text
resources/js/Layouts/AppLayout.tsx
```

布局负责所有业务页面共有的东西：

- 顶部品牌。
- 主导航。
- 当前登录人展示。
- 退出表单。
- 页面标题区域。

关键代码：

```tsx
export default function AppLayout({ activeNav, children, subtitle, title }: AppLayoutProps) {
    // usePage 可以读取 Laravel 通过 Inertia 共享的 props。
    const { auth } = usePage<SharedProps>().props;

    return (
        <main>
            <header>
                {/* 所有业务页面共用导航 */}
            </header>

            <section>
                <h1>{title}</h1>
                {children}
            </section>
        </main>
    );
}
```

为什么 React 页面可以放在 Laravel 项目里：

- Laravel 负责路由、认证、查询数据库。
- Inertia 把 Laravel 的页面名和数据交给前端。
- React 负责渲染交互界面。
- Vite 负责编译 React/TypeScript。

它不是传统 Blade，也不是完全前后端分离，而是单体项目内的现代前端页面。

## 第四步：任务大厅 React 页面

文件：

```text
resources/js/Pages/Tasks/Index.tsx
```

页面接收 Controller 传来的 props：

```tsx
type TaskIndexProps = {
    complexityOptions: SelectOption[];
    filters: TaskFilters;
    statusOptions: SelectOption[];
    tasks: PaginatedTasks;
};
```

对应 Laravel：

```php
return Inertia::render('Tasks/Index', [
    'filters' => $filters,
    'statusOptions' => $this->statusOptions(),
    'complexityOptions' => $this->complexityOptions(),
    'tasks' => [...],
]);
```

字段名必须一致。Laravel 传 `filters`，React 就用 `filters`；Laravel 传 `tasks.data`，React 就从 `tasks.data` 渲染列表。

筛选使用 GET 表单：

```tsx
<form action="/tasks" method="GET">
    <input name="keyword" />
    <select name="status" />
    <select name="complexity" />
    <button type="submit">查询</button>
</form>
```

为什么使用 GET：

- 筛选不会修改服务器数据。
- URL 可以分享，例如 `/tasks?status=OPEN&keyword=登录`。
- 刷新页面不会丢失筛选条件。
- 后端可以直接从 Query String 读取条件。

## 第五步：退出仍然使用原生 POST Form

`AppLayout` 中退出按钮仍然是：

```tsx
<form action="/logout" method="POST">
    <input name="_token" type="hidden" value={csrfToken()} />
    <button type="submit">退出</button>
</form>
```

原因：

- `/logout` 会修改 Session，所以必须是 POST。
- 后端可能返回公司 SSO 外部退出地址。
- 如果用 Inertia Ajax 跟随外部 `302`，会触发浏览器 CORS。
- 原生 Form 会触发浏览器顶层导航，不走跨域 XHR。

## 第六步：补充 Feature Test

文件：

```text
tests/Feature/ApplicationShellTest.php
```

测试内创建最小 `task` 和 `bid` 表，是因为当前项目不执行 Laravel Migration，测试数据库默认是内存 SQLite。

关键断言：

```php
$this->withSession([...])
    ->withHeader('X-Inertia', 'true')
    ->withHeader('X-Inertia-Version', inertiaVersionForTest())
    ->get('/tasks?status=PENDING_SELECTION&complexity=MEDIUM&keyword=%E7%99%BB%E5%BD%95')
    ->assertOk()
    ->assertJsonPath('component', 'Tasks/Index')
    ->assertJsonPath('props.filters.status', 'PENDING_SELECTION')
    ->assertJsonPath('props.tasks.data.0.displayStatus', 'PENDING_SELECTION');
```

这个测试验证的是：

- 已登录用户可以访问任务大厅。
- Query String 能进入 Controller。
- Controller 能筛出“待选标”任务。
- Inertia 页面名和 props 正确。

这里加 `X-Inertia` 请求头，是为了让 Laravel 返回 Inertia JSON 响应，方便用 `assertJsonPath()` 直接检查组件名和 props。`X-Inertia-Version` 用于模拟 Inertia 客户端的资源版本协商；真实浏览器由 Inertia 自动处理。当前测试辅助函数按 `public/build/manifest.json` 计算版本，和 Inertia Middleware 的默认规则保持一致。第一次访问页面时浏览器会拿到 HTML，后续 Inertia 页面切换才会拿 JSON。

测试 URL 中的 `keyword=%E7%99%BB%E5%BD%95` 是“登录”的 URL 编码形式。浏览器地址栏里输入中文时会自动编码；测试代码里建议直接写编码后的 Query String，避免 JSON 响应中出现未编码 URL。

## 常见错误

| 错误 | 原因 | 处理 |
|---|---|---|
| `/tasks` 跳转 `/login` | 当前没有 SSO Session | 真实登录或使用测试验证 |
| `Table task does not exist` | 本地没有执行 `database/schema.sql` | 按第 04 章初始化数据库 |
| `Property does not exist` | Laravel props 和 React TypeScript 类型不一致 | 对照 `TaskIndexProps` 和 Controller 返回字段 |
| 筛选后分页丢条件 | 没有调用 `withQueryString()` | Controller 分页查询保留 `withQueryString()` |
| 退出触发 CORS | 使用了 Inertia Ajax 退出 | 改回原生 POST Form |
