# 01-项目初始化

## 1. 本章节目标

本章说明 TaskHub Phase 1 如何从空仓库初始化为 Laravel 13 单体项目。目标不是开发业务功能，而是建立能运行、能测试、能继续扩展的基础工程。

对 Java 开发者来说，可以把这一步理解为创建一个 Spring Boot 项目骨架：确定运行入口、依赖管理、环境变量、测试命令和本地开发命令。

## 2. 涉及文件

| 文件 | 作用 |
|---|---|
| `composer.json` / `composer.lock` | PHP 依赖管理，类似 Maven `pom.xml` 或 Gradle `build.gradle` |
| `package.json` / `package-lock.json` | 前端依赖管理，服务 React、TypeScript、Vite |
| `artisan` | Laravel 命令行入口，类似 Spring Boot 项目的 Maven/Gradle task 加一部分框架 CLI |
| `.env.example` | 环境变量模板，不保存真实密钥 |
| `bootstrap/app.php` | Laravel 13 应用启动配置入口 |
| `config/*.php` | Laravel 配置文件，对应 Spring Boot 的 `application.yml` 加配置类 |

## 3. 实现步骤

1. 使用 Composer 创建 Laravel 13 骨架。
2. 将项目 PHP 版本约束调整为 `^8.5.8`。
3. 安装 Inertia Laravel 适配器。
4. 安装 Pest 核心测试框架。
5. 移除 Laravel 默认会执行 migration 的脚本，避免误改已有数据库。
6. 将开发脚本调整为只启动 Laravel 服务、日志和 Vite，不启动 queue worker。

## 4. 核心代码讲解

`composer.json` 是 PHP 世界的依赖入口。`require` 类似 Java 的 production dependencies，`require-dev` 类似 testImplementation / developmentOnly。

`bootstrap/app.php` 是 Laravel 13 的启动配置文件，用链式 API 注册路由、中间件、异常处理。它承担了部分 Spring Boot `@SpringBootApplication` 和 WebMvcConfigurer 的职责。

`artisan` 是 Laravel 的 CLI。常用命令：

```bash
php artisan serve
php artisan route:list
php artisan config:clear
```

## 5. 设计原因

本项目采用 Laravel Monolith + React(Inertia)，原因是：

- 后端、前端、路由和部署在一个项目内，MVP 开发成本低。
- Inertia 保留服务端路由体验，但页面用 React 写，不退回传统 Blade。
- 不做完全前后端分离，避免过早引入 API 鉴权、跨域、前后端双部署。
- 不引入 Repository Pattern，因为 Eloquent Model 已经覆盖当前阶段的数据映射需求。

Pest Laravel 插件当前未采用，因为当前可用版本的约束还未覆盖 Laravel 13；本项目安装 Pest 核心并通过 `tests/Pest.php` 绑定 Laravel `Tests\TestCase`。

## 6. 注意事项

- 不要运行 `php artisan migrate` 修改现有数据库。
- `.env` 不提交，提交的是 `.env.example`。
- `composer install` 安装 PHP 依赖，`npm install` 安装前端依赖。
- Laravel 默认生成的 users/cache/jobs migration 已删除。

## 7. 本阶段完成效果

完成后应能：

```bash
composer install
npm install
php artisan about
```

项目可以被 Laravel 正常识别，依赖可以安装，命令行入口可用。
