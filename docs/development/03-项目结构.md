# 03-项目结构

## 1. 本章目标

理解 TaskHub 为什么保持 Laravel 官方目录，并知道以后新代码应该放在哪里。

## 2. 前置条件

已经拉取或创建 TaskHub 项目。

## 3. 本章最终效果

你能判断 Model、Service、SSO 集成、Inertia 页面、配置和测试分别在哪个目录。

## 4. 实际执行命令

```bash
find app config resources routes tests docs/development -maxdepth 3 -type f | sort
```

Windows PowerShell：

```powershell
Get-ChildItem app,config,resources,routes,tests,docs/development -Recurse -File
```

## 5. 命令执行目录

项目根目录。

## 6. 命令作用

查看当前项目实际文件结构，避免只看理论目录。

## 7. 预期输出或生成文件

你会看到：

```text
app/Models/Task.php
app/Services/CurrentUserService.php
app/Integrations/Sso/SsoClient.php
resources/js/app.tsx
routes/web.php
```

本章不生成文件。

## 8. 逐文件修改过程

本章以阅读为主。新增代码时遵守：

- 数据表映射放 `app/Models`。
- 当前用户获取放 `app/Services/CurrentUserService.php`。
- 公司 SSO 调用放 `app/Integrations/Sso`。
- HTTP 中间件放 `app/Http/Middleware`。
- React 页面放 `resources/js/Pages`。
- 文档放 `docs/development`。

不要新增：

- `Repository`
- `Domain`
- `Application`
- `Infrastructure`

## 9. 修改前后的关键代码

无代码修改。以下是结构判断示例：

```text
app/Models/Task.php                 # 表映射
app/Services/CurrentUserService.php # 当前用户统一入口
routes/web.php                      # 页面路由
```

## 10. 核心原理解释

Laravel 官方目录已经覆盖 MVC、配置、路由、测试、资源编译。Phase 1 不需要额外分层。过早分层会增加 Java 开发者学习成本，也会把 Eloquent 的直接查询能力包装掉。

## 11. 与 Spring Boot 的概念对照

| Laravel 目录 | Spring Boot 类比 |
|---|---|
| `routes/web.php` | Controller 路由映射 |
| `app/Models` | JPA Entity + Active Record |
| `app/Services` | `@Service` |
| `config` | `application.yml` + 配置类 |
| `app/Http/Middleware` | Filter / Interceptor |
| `resources/js` | React 前端模块 |

## 12. 验证步骤

```bash
php artisan route:list
php artisan test
```

确认路由和测试仍正常。

## 13. 常见错误与解决方法

| 错误 | 处理 |
|---|---|
| 不知道 Service 放哪里 | 放 `app/Services` |
| 想建 Repository | 先使用 Eloquent，确有重复复杂查询再讨论 |
| 想建 Domain/Application | Phase 1 禁止，保持官方目录 |
| React 页面找不到 | 放到 `resources/js/Pages` 并检查文件名 |
