# 03-项目结构

## 1. 本章目标

本章不是简单背目录，而是帮助你建立 Laravel 项目的整体地图。

学完以后，你应该能回答：

- PHP / Laravel 项目为什么有这些目录。
- 浏览器访问 TaskHub 首页时，实际经过了哪些文件。
- Laravel 是从哪里启动的。
- `Route`、`Controller`、`Model`、`Middleware`、`Blade`、`React Page` 分别负责什么。
- 为什么 React 页面可以放在 Laravel 项目里。
- Inertia 到底解决了什么问题。
- 一个熟悉 Java / Spring Boot 的开发者，应该如何理解 Laravel 的组织方式。

本章只阅读和理解当前项目结构，不开发任务发布、投标、交付等业务功能。

## 2. 学习目标

完成本章后，你需要重点掌握：

- `public/index.php` 是浏览器请求进入 Laravel 的入口。
- `bootstrap/app.php` 负责组装 Laravel 应用、路由和 middleware。
- `routes/web.php` 负责定义浏览器页面路由。
- `resources/views/app.blade.php` 是 Inertia + React 的 HTML 容器。
- `resources/js/app.tsx` 是 React/Inertia 的前端启动入口。
- `resources/js/Pages` 存放 React 页面。
- `app/Models` 中的 Eloquent Model 映射数据库表。
- `vendor` 和 `node_modules` 都是依赖目录，不手工修改。

## 6. 每一步操作

### 6.1 先看根目录

项目根目录可以理解为 Java 项目的工程根目录，类似包含 `pom.xml` 或 `build.gradle` 的目录。

TaskHub 的关键根目录结构：

```text
task-hub/
├── artisan
├── app/
├── bootstrap/
├── config/
├── database/
├── docs/
├── public/
├── resources/
├── routes/
├── storage/
├── tests/
├── vendor/
├── node_modules/
├── composer.json
├── package.json
├── tsconfig.json
└── vite.config.ts
```

你平时主要修改：

- `app/`：后端 PHP 代码。
- `routes/`：路由。
- `resources/`：Blade、React、CSS。
- `config/`：配置。
- `database/schema.sql`：当前数据库结构标准。
- `tests/`：测试。
- `docs/development/`：开发文档。

通常不手工修改：

- `vendor/`：Composer 安装的 PHP 依赖。
- `node_modules/`：npm 安装的前端依赖。
- `storage/framework/`：Laravel 运行缓存。

### 6.2 理解 `artisan`

文件：`artisan`

`artisan` 是 Laravel 的命令行入口，类似 Spring Boot 项目中通过 Maven/Gradle 执行的命令集合。

常见命令：

```bash
php artisan route:list
php artisan about
php artisan test
php artisan config:clear
```

开发过程中什么时候修改：

- 几乎不修改 `artisan` 文件本身。
- 你只是通过它执行 Laravel 命令。

与 Spring Boot 的相似点：

- 都是开发、调试、检查项目的命令入口。

不同点：

- Spring Boot 常通过 `mvn`、`gradle` 或 IDE 启动。
- Laravel 使用 `php artisan ...` 作为统一命令入口。

### 6.3 理解 `public/index.php`

文件：`public/index.php`

这是所有浏览器请求进入 Laravel 的入口文件，也叫 Front Controller。

当前关键代码：

```php
<?php

use Illuminate\Foundation\Application;
use Illuminate\Http\Request;

// 记录请求开始时间，Laravel 会用它计算请求耗时。
define('LARAVEL_START', microtime(true));

// 如果项目处于维护模式，会先加载维护页面。
if (file_exists($maintenance = __DIR__.'/../storage/framework/maintenance.php')) {
    require $maintenance;
}

// 加载 Composer 自动加载器。
// 这样 App\Models\Task 等 PHP 类才能按命名空间自动找到。
require __DIR__.'/../vendor/autoload.php';

// 加载 Laravel 应用配置，并得到 Application 对象。
/** @var Application $app */
$app = require_once __DIR__.'/../bootstrap/app.php';

// 捕获当前 HTTP 请求，并交给 Laravel 处理。
$app->handleRequest(Request::capture());
```

为什么浏览器会进入这里：

- Web 服务器的公开目录是 `public/`。
- 浏览器访问 `/` 时，入口会落到 `public/index.php`。
- Laravel 不希望浏览器直接访问 `app/`、`config/`、`database/` 等源码目录。

开发过程中什么时候修改：

- 正常业务开发几乎不修改。
- 只有非常底层的启动逻辑才会碰它。

与 Spring Boot 的相似点：

- 类似 Spring Boot 内部接收 HTTP 请求的入口。

不同点：

- Spring Boot 的入口通常是 `main()` 方法和内嵌服务器。
- PHP Web 项目由 Web 服务器把请求交给 `public/index.php`。

### 6.4 理解 `bootstrap/app.php`

文件：`bootstrap/app.php`

这个文件负责组装 Laravel 应用：路由在哪里、middleware 如何注册、异常如何处理。

当前关键代码：

```php
<?php

use Illuminate\Foundation\Application;
use Illuminate\Foundation\Configuration\Exceptions;
use Illuminate\Foundation\Configuration\Middleware;
use Illuminate\Http\Request;
use App\Http\Middleware\HandleInertiaRequests;

return Application::configure(basePath: dirname(__DIR__))
    ->withRouting(
        // 注册 Web 路由文件。
        // 浏览器页面请求会进入 routes/web.php。
        web: __DIR__.'/../routes/web.php',

        // 注册命令行路由文件。
        commands: __DIR__.'/../routes/console.php',

        // Laravel 健康检查地址。
        health: '/up',
    )
    ->withMiddleware(function (Middleware $middleware): void {
        // 将 Inertia middleware 追加到 web middleware 组。
        // 这样所有 Web 页面都能使用 app.blade.php 作为根视图。
        $middleware->web(append: [
            HandleInertiaRequests::class,
        ]);
    })
    ->withExceptions(function (Exceptions $exceptions): void {
        // api/* 请求返回 JSON 异常响应。
        // Web 页面异常仍按 Laravel 页面方式处理。
        $exceptions->shouldRenderJsonWhen(
            fn (Request $request) => $request->is('api/*'),
        );
    })->create();
```

为什么下一步会进入 `routes/web.php`：

- `bootstrap/app.php` 中 `withRouting(web: ...)` 明确告诉 Laravel：Web 页面路由定义在 `routes/web.php`。

开发过程中什么时候修改：

- 注册全局 middleware。
- 调整路由文件。
- 配置全局异常处理。

与 Spring Boot 的相似点：

- 类似 Spring Boot 自动配置和 Web 配置类的一部分。

不同点：

- Laravel 13 把启动配置集中在 `bootstrap/app.php`。
- Spring Boot 更依赖注解扫描、自动配置和 Bean 容器。

### 6.5 理解 `routes/web.php`

文件：`routes/web.php`

这个文件定义浏览器访问页面时的路由。

当前完整代码：

```php
<?php

use Inertia\Inertia;
use Illuminate\Support\Facades\Route;

// 注册首页路由。
// 当浏览器访问 "/" 时，会执行下面这个闭包。
Route::get('/', function () {
    // 返回 React 页面 Welcome。
    // Welcome 实际对应 resources/js/Pages/Welcome.tsx。
    return Inertia::render('Welcome', [
        // 将 Laravel 配置中的应用名称传递给 React。
        'appName' => config('app.name'),
    ]);
});
```

为什么当前首页没有 Controller：

- Phase 1 首页只是验证 Laravel + Inertia + React 是否连通。
- 逻辑很少，直接写闭包更简单。
- 后续出现任务列表、投标、交付等业务时，会改为 Controller。

后续业务路由可能长这样：

```php
<?php

use App\Http\Controllers\TaskController;
use Illuminate\Support\Facades\Route;

// 示例：后续业务页面会交给 Controller。
// 当前项目还没有实现这个业务路由。
Route::get('/tasks', [TaskController::class, 'index']);
```

与 Spring Boot 的相似点：

- `Route::get('/tasks', ...)` 类似 `@GetMapping("/tasks")`。

不同点：

- Laravel 可以在路由文件中直接写闭包。
- Spring Boot 通常把路由写在 Controller 注解上。

### 6.6 理解 Controller

当前文件：`app/Http/Controllers/Controller.php`

当前项目只有 Laravel 默认基础 Controller，还没有业务 Controller。

Controller 的职责：

- 接收路由分发过来的请求。
- 调用 Service 或 Model 获取数据。
- 返回 Inertia 页面、JSON 或重定向。

后续业务页面示例：

```php
<?php

namespace App\Http\Controllers;

use App\Models\Task;
use Inertia\Inertia;

class TaskController extends Controller
{
    public function index()
    {
        // 示例：查询任务表。
        // 当前文档只说明结构，不新增此业务代码。
        $tasks = Task::query()
            ->latest('created_at')
            ->limit(10)
            ->get();

        // 将查询结果作为 props 传给 React 页面。
        return Inertia::render('Tasks/Index', [
            'tasks' => $tasks,
        ]);
    }
}
```

开发过程中什么时候修改：

- 新增业务页面。
- 处理表单提交。
- 做请求协调和页面返回。

与 Spring Boot 的相似点：

- 类似 `@Controller` 或 `@RestController`。

不同点：

- Laravel Controller 可以直接返回 `Inertia::render()`。
- Spring Boot 常返回 JSON、模板名或 `ModelAndView`。

### 6.7 理解 Model 和 MySQL

目录：`app/Models`

TaskHub 当前已有业务 Model：

```text
app/Models/Task.php
app/Models/Bid.php
app/Models/BidMember.php
app/Models/TaskAssignee.php
app/Models/TaskDelivery.php
app/Models/AttachmentRef.php
app/Models/TaskChangeRequest.php
app/Models/TaskEvent.php
```

以 `Task` 为例：

```php
<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Model;

class Task extends Model
{
    // 指定数据库表名。
    // TaskHub 的表名是单数 task，不使用 Laravel 默认复数 tasks。
    protected $table = 'task';

    // 当前数据库主键由外部或业务侧生成，不使用数据库自增。
    public $incrementing = false;

    // 主键类型仍是整数。
    protected $keyType = 'int';

    // 允许批量写入的字段。
    // 这不是业务校验，只是防止随意写入未声明字段。
    protected $fillable = [
        'id',
        'title',
        'description',
        'status',
        'created_by',
    ];
}
```

上面代码为了讲解只展示关键字段，实际 `Task.php` 中字段更多，并包含 casts 和 relationships。

Model 的职责：

- 映射数据库表。
- 定义字段类型转换。
- 定义表之间的关联关系。
- 通过 Eloquent 查询数据库。

开发过程中什么时候修改：

- 新增表对应 Model。
- 表字段变化后同步 `$fillable`、`casts()`。
- 增加表关系，例如 `hasMany`、`belongsTo`。

与 Spring Boot 的相似点：

- 类似 JPA Entity。

不同点：

- Eloquent 是 Active Record 风格，Model 自己带查询能力。
- JPA Entity 通常只描述数据，查询常放在 Repository。
- TaskHub Phase 1 不引入 Repository，优先使用 Laravel 官方 Eloquent。

### 6.8 理解 `config/`

目录：`config`

当前关键配置：

```text
config/app.php
config/database.php
config/session.php
config/auth.php
config/sso.php
```

职责：

- 读取 `.env` 中的环境变量。
- 组织 Laravel 运行配置。
- 让代码通过 `config('app.name')` 读取配置。

例如首页路由中：

```php
// 从 config/app.php 读取应用名。
'appName' => config('app.name')
```

开发过程中什么时候修改：

- 新增 SSO、外部服务、文件服务等配置项。
- 调整数据库、Session、日志等框架配置。

与 Spring Boot 的相似点：

- 类似 `application.yml`、`application.properties` 和配置类。

不同点：

- Laravel 配置是 PHP 数组文件。
- Spring Boot 配置通常是 YAML/Properties，再绑定到 Bean。

### 6.9 理解 `database/`

目录：`database`

TaskHub 当前重点文件：

```text
database/schema.sql
database/seeders/DatabaseSeeder.php
```

职责：

- `database/schema.sql` 是当前数据库结构标准。
- `seeders` 可用于准备测试或演示数据，但当前 Phase 1 尚未使用。

重要原则：

- 当前不执行默认 Laravel migration 创建业务结构。
- 不创建本地 `users` 表。
- 业务表结构以 `database/schema.sql` 为准。

与 Spring Boot 的相似点：

- 类似 `schema.sql`、Flyway、Liquibase 的数据库脚本管理。

不同点：

- Laravel 官方常用 migration。
- TaskHub 当前为了贴合已确认的数据库设计，先以 `database/schema.sql` 为唯一标准。

### 6.10 理解 `resources/`

目录：`resources`

当前结构：

```text
resources/
├── css/
│   └── app.css
├── js/
│   ├── app.tsx
│   ├── vite-env.d.ts
│   └── Pages/
│       └── Welcome.tsx
└── views/
    └── app.blade.php
```

职责：

- `resources/views/app.blade.php`：HTML 容器。
- `resources/js/app.tsx`：React/Inertia 启动入口。
- `resources/js/Pages`：React 页面。
- `resources/css/app.css`：全局样式和 Tailwind 配置入口。

为什么 React 页面能放在 Laravel 项目里：

- TaskHub 是 Laravel Monolith，不是前后端分离项目。
- Laravel 负责返回页面入口和服务端能力。
- React 负责浏览器内的交互和渲染。
- Vite 将 `resources/js` 和 `resources/css` 编译成浏览器可加载的静态资源。

与 Spring Boot 的相似点：

- 类似 `src/main/resources/templates` 和前端资源目录。

不同点：

- TaskHub 的业务页面不是 Blade 模板，而是 React TSX。
- Blade 只负责放置 `@vite` 和 `@inertia` 容器。

### 6.11 理解 `storage/`

目录：`storage`

职责：

- 存放日志。
- 存放缓存。
- 存放编译后的 Blade 视图缓存。
- 存放临时文件和运行时文件。

开发过程中什么时候修改：

- 通常不手工修改。
- 排查日志时查看 `storage/logs/laravel.log`。


与 Spring Boot 的相似点：

### 6.13 理解 `vendor/`

目录：`vendor`

职责：

- Composer 安装的 PHP 依赖。
- Laravel 框架代码也在这里。

开发过程中什么时候修改：

- 不手工修改。
- 通过 `composer require`、`composer install`、`composer update` 管理。

与 Spring Boot 的相似点：

不同点：

- PHP 项目通常把依赖安装到项目内的 `vendor/`。
- Java Maven 依赖通常在本地仓库，构建时进入 classpath。

职责：

- React、Vite、TypeScript、Tailwind 都在这里。

开发过程中什么时候修改：

- 不手工修改。

与 Spring Boot 的相似点：

- 类似前端构建依赖目录。

不同点：

## 7. 每一步为什么这么做

TaskHub 当前坚持 Laravel 官方目录，是为了降低学习和维护成本。

不新增复杂分层的原因：

- Laravel 官方目录已经覆盖 MVC、配置、路由、测试、资源编译。
- Eloquent 已经能表达简单 CRUD 和关联查询。
- Phase 1 目标是建立基础架构和学习路径，不是搭建大型架构模板。
- 过早加入 Repository、DDD、Application、Infrastructure 会让 Java 开发者误以为 Laravel 必须按 Java 分层方式编写。

React 放在 Laravel 项目里的原因：

- TaskHub 采用 Laravel Monolith + Inertia。
- 后端和前端在一个项目中构建、部署、维护。
- 不需要维护独立 API 项目和独立前端项目。
- 仍然可以使用 React + TypeScript 构建现代交互页面。

Inertia 解决的问题：

- 传统 Laravel Blade 页面交互能力有限。
- 完全前后端分离又需要额外维护 API、跨域、Token、前后端部署。
- Inertia 让 Laravel 路由直接返回 React 页面名和 props。
- 这样既保留 Laravel 的服务端能力，又使用 React 写页面。

## 8. 当前首页请求流程

当前首页真实流程如下：

```text
┌──────────────┐
│ 浏览器访问 / │
└──────┬───────┘
       ↓
┌──────────────────┐
│ public/index.php │
│ Laravel 入口文件 │
└──────┬───────────┘
       ↓
┌──────────────────┐
│ bootstrap/app.php│
│ 注册路由和中间件 │
└──────┬───────────┘
       ↓
┌──────────────────┐
│ routes/web.php   │
│ 匹配 GET /       │
└──────┬───────────┘
       ↓
┌──────────────────────────┐
│ Inertia::render('Welcome')│
│ 返回页面名和 props        │
└──────┬───────────────────┘
       ↓
┌──────────────────────────────┐
│ HandleInertiaRequests        │
│ 指定 rootView = app          │
└──────┬───────────────────────┘
       ↓
┌──────────────────────────────┐
│ resources/views/app.blade.php│
│ 输出 HTML 容器               │
└──────┬───────────────────────┘
       ↓
┌────────────────────┐
│ resources/js/app.tsx│
│ 启动 React/Inertia │
└──────┬─────────────┘
       ↓
┌───────────────────────────────┐
│ resources/js/Pages/Welcome.tsx│
│ 渲染 React 页面               │
└──────┬────────────────────────┘
       ↓
┌──────────────┐
│ 浏览器显示页面│
└──────────────┘
```

逐步解释：

1. 浏览器访问 `/`，Web 服务器将请求交给 `public/index.php`。
2. `public/index.php` 加载 `vendor/autoload.php`，让 PHP 类可以自动加载。
3. `public/index.php` 加载 `bootstrap/app.php`，得到 Laravel Application。
4. `bootstrap/app.php` 注册 `routes/web.php` 和 `HandleInertiaRequests`。
5. Laravel 根据 `routes/web.php` 找到 `GET /`。
6. 路由返回 `Inertia::render('Welcome', ['appName' => config('app.name')])`。
7. Inertia 根据 middleware 的 `$rootView = 'app'` 使用 `resources/views/app.blade.php`。
8. Blade 中的 `@vite('resources/js/app.tsx')` 加载 React 入口。
9. `app.tsx` 通过 `import.meta.glob('./Pages/**/*.tsx')` 找到 `Welcome.tsx`。
10. React 接收 `appName` props 并渲染页面。

## 9. 后续业务页面完整请求流程

后续当我们开发一个真正业务页面，例如“任务列表”时，请求会多经过 Controller、Model 和 MySQL。

完整业务链路会是：

```text
浏览器
↓
public/index.php
↓
bootstrap/app.php
↓
routes/web.php
↓
Controller
↓
Model
↓
MySQL
↓
Controller 组织 props
↓
Inertia::render()
↓
resources/views/app.blade.php
↓
resources/js/app.tsx
↓
React Page
↓
浏览器
```

每一步职责：

| 步骤 | 为什么进入这里 | 负责什么 | 下一步为什么进入那里 |
|---|---|---|---|
| 浏览器 | 用户访问页面 URL | 发起 HTTP 请求 | Web 服务器公开目录指向 `public/` |
| `public/index.php` | Laravel 统一入口 | 加载依赖并捕获请求 | 需要创建 Laravel Application |
| `bootstrap/app.php` | 入口文件加载它 | 注册路由、中间件、异常处理 | 根据路由配置进入 `routes/web.php` |
| `routes/web.php` | Web 路由定义在这里 | 匹配 URL 和 HTTP 方法 | 复杂业务交给 Controller |
| Controller | 路由指定 Controller 方法 | 协调请求、查询数据、返回页面 | 需要数据时调用 Model |
| Model | Controller 查询业务数据 | 映射表并执行 Eloquent 查询 | Eloquent 根据数据库配置访问 MySQL |
| MySQL | Model 发起 SQL | 返回业务数据 | 数据回到 Controller |
| `Inertia::render()` | Controller 返回页面 | 指定 React 页面名和 props | Inertia 使用根 Blade |
| `app.blade.php` | Inertia rootView 指向它 | 输出 HTML 容器和 Vite 入口 | 浏览器加载 `app.tsx` |
| `app.tsx` | Vite 入口 | 启动 React 并解析页面名 | 找到 React Page |
| React Page | 页面名映射到 TSX 文件 | 渲染用户界面 | 浏览器显示页面 |

## 10. Laravel + React + Inertia 架构图

```text
┌────────────────────────────────────────────────────┐
│                    浏览器                           │
│  HTML 容器 + Vite JS + React 页面                    │
└──────────────────────┬─────────────────────────────┘
                       │
                       │ HTTP 请求 / Inertia 访问
                       ↓
┌────────────────────────────────────────────────────┐
│                  Laravel Monolith                  │
│                                                    │
│  public/index.php                                  │
│        ↓                                           │
│  bootstrap/app.php                                 │
│        ↓                                           │
│  routes/web.php                                    │
│        ↓                                           │
│  Controller / Middleware / Service                 │
│        ↓                                           │
│  Eloquent Model                                    │
│        ↓                                           │
│  MySQL                                             │
│                                                    │
│  Inertia::render(Page, Props)                      │
└──────────────────────┬─────────────────────────────┘
                       │
                       │ 页面名 + props
                       ↓
┌────────────────────────────────────────────────────┐
│              resources/views/app.blade.php         │
│  @vite('resources/js/app.tsx') + @inertia           │
└──────────────────────┬─────────────────────────────┘
                       │
                       ↓
┌────────────────────────────────────────────────────┐
│                  React + TypeScript                 │
│  resources/js/app.tsx                               │
│        ↓                                           │
│  resources/js/Pages/*.tsx                           │
└────────────────────────────────────────────────────┘
```

这张图的关键点：

- Laravel 和 React 在同一个仓库中。
- Laravel 控制路由、配置、认证、数据库。
- React 控制页面交互和渲染。
- Inertia 是两者之间的桥，不是消息队列，也不是 API 网关。

## 11. Laravel 与 Spring Boot 对照

| Laravel | Spring Boot | 相似点 | 不能完全对应的地方 |
|---|---|---|---|
| `routes/web.php` / Route | `@RequestMapping` / `@GetMapping` | 都定义 URL 到处理逻辑的映射 | Laravel 路由集中写在文件里，也可以直接写闭包 |
| Middleware | Filter / Interceptor | 都能在请求前后处理逻辑 | Laravel middleware 按 `web`、`api` 等组注册 |
| Controller | Controller | 都负责处理请求和返回响应 | Laravel 可直接返回 `Inertia::render()` |
| Model / Eloquent | Entity / JPA | 都描述业务数据和数据库表 | Eloquent 是 Active Record，Model 自带查询能力 |
| Service | Service | 都承载可复用业务逻辑 | Laravel 不强制所有业务都必须放 Service |
| `config/*.php` | `application.yml` / 配置类 | 都管理应用配置 | Laravel 配置是 PHP 数组 |
| `.env` | 环境变量 / profile 配置 | 都用于环境差异配置 | `.env` 不提交真实敏感值 |
| `artisan` | Maven / Gradle 命令、Spring Boot CLI | 都用于运行项目命令 | artisan 是 Laravel 自带命令系统 |
| `composer` | Maven / Gradle | 都管理后端依赖 | Composer 管 PHP 包，依赖安装到 `vendor/` |
| `npm` | npm / pnpm / yarn | 都管理前端依赖 | 与 Spring Boot 本身无关，但现代 Java 项目也常用 |
| Blade | Thymeleaf / JSP | 都能生成 HTML | TaskHub 只用 Blade 做 React 容器 |
| Inertia Props | ViewModel / Model attributes | 都把服务端数据传给页面 | Inertia props 直接进入 React 组件 |
| Pest | JUnit | 都写自动化测试 | Pest 语法更偏函数式 |

## 13. 常见错误

| 错误 | 常见原因 | 处理 |
|---|---|---|
| 不知道新 PHP 类放哪里 | 没有先判断职责 | 表映射放 `app/Models`，业务协调放 Controller，可复用逻辑放 `app/Services` |
| 想创建 `Repository` | 用 Java 分层习惯套 Laravel | Phase 1 优先使用 Eloquent，确实复杂后再评估 |
| React 页面找不到 | 页面名和文件路径不一致 | `Inertia::render('Welcome')` 对应 `resources/js/Pages/Welcome.tsx` |
| 修改 Blade 后没变化 | 视图缓存未清理 | 执行 `php artisan view:clear` |
| 修改配置后没变化 | 配置缓存未清理 | 执行 `php artisan config:clear` |
| `vendor` 目录缺失 | 没有安装 PHP 依赖 | 执行 `composer install` |
| `node_modules` 缺失 | 没有安装前端依赖 | 执行 `npm install` |
| 页面 500 | Laravel 后端异常 | 查看 `storage/logs/laravel.log` |
| Vite manifest 错误 | 没启动 Vite 或没构建 | 开发运行 `composer run dev`，构建运行 `npm run build` |

## 14. 本章总结

TaskHub 当前结构可以用一句话理解：

```text
Laravel 负责请求、路由、配置、认证、数据库；
Inertia 负责把 Laravel 页面响应连接到 React；
React 负责浏览器里的页面展示和交互；
Vite 负责前端开发和构建。
```

当前首页请求没有 Controller、Model、MySQL，是因为它只是基础架构验证页。

后续真正业务页面会逐步加入：

- Controller：接收请求并组织页面数据。
- Model：查询 MySQL。
- Inertia Props：把数据传给 React。
- React Page：展示业务数据。
