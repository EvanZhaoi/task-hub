# 12-发布任务模态框

## 本章目标

本章开始第一个正式写入业务：在任务大厅中通过模态框发布招标任务。

本章不创建单独的 `/tasks/create` 页面。发布任务作为任务大厅里的高频操作，直接使用模态框承载。

## 学习目标

- React 如何用 shadcn/Radix Dialog 做模态框。
- Inertia `useForm()` 如何提交 Laravel Web 表单。
- Laravel FormRequest 如何做后端校验。
- Eloquent 如何在一个事务中写入多张表。
- 为什么附件只记录外部 `attachment_id`。
- 为什么任务发布要同时写 `task_event`。

## 最终效果

访问：

```text
http://127.0.0.1:8000/tasks
```

点击右上角“发布任务”，填写表单并提交后：

- 新增一条 `task`。
- 任务状态为 `OPEN`。
- 分配方式为 `BIDDING`。
- 后端根据付款账号 ID 调外部接口，保存付款账号历史快照。
- 可选附件 ID 写入 `attachment_ref`。
- 写入 `TASK_CREATED` 和 `TASK_PUBLISHED` 两条 `task_event`。
- 页面返回任务大厅并显示“任务已发布。”


外部付款账号接口配置写入 `.env`。真实地址、path 和请求方式以公司接口为准：

```env
# 外部付款账号服务基础地址，例如 https://payment.example.com。
PAYMENT_ACCOUNT_BASE_URL=

# 查询账号详情的 path，只写 path，不写完整 URL。
# 支持 /accounts/{accountId}，也支持 /accounts 这种 query string 方式。

# 当前客户端支持 GET 和 POST；不知道真实协议时先不要猜，等接口确认后填写。
PAYMENT_ACCOUNT_METHOD=GET

# 网络请求超时时间，避免发布任务请求长时间等待外部服务。
PAYMENT_ACCOUNT_TIMEOUT=3

# 内网测试环境如果证书不完整可先 false；生产环境建议 true。
PAYMENT_ACCOUNT_VERIFY_SSL=false
```

## 每一步操作

### 1. 新增 FormRequest

文件：

```text
app/Http/Requests/StoreTaskRequest.php
```

关键点：

- `authorize()` 返回 `true`，因为登录校验由 `EnsureSsoAuthenticated` 负责。
- 字段名使用前端 camelCase，例如 `expectedDelivery`、`biddingDeadline`。
- 后端保存时再映射成数据库字段 `expected_delivery`、`bidding_deadline`。
- `attachmentIds` 是字符串，允许换行、逗号、空格分隔多个外部附件 ID。

### 2. 新增 SnowflakeId 服务

文件：

```text
app/Services/SnowflakeId.php
```

为什么需要它：

- schema.sql 中业务主键是 `BIGINT UNSIGNED`。
- Model 都设置了 `$incrementing = false`。
- 当前阶段不使用 MySQL 自增。
- 暂未接入公司统一 ID 服务。

当前实现是本地轻量方案，后续如果公司提供统一雪花 ID 服务，只替换这个服务。

当前 ID 结构：

```text
当前毫秒时间戳 * 1000 + 同毫秒内自增序号
```

说明：

- `currentMillis()` 返回当前毫秒时间戳。
- `sequence` 表示同一毫秒内的自增序号，范围是 `0~999`。
- 一个毫秒内最多生成 1000 个 ID。
- 如果同一毫秒内超过 1000 个 ID，就等待下一毫秒。
- 最终 ID 仍然是 BIGINT 整数，适合写入当前 schema.sql 中的主键字段。

为什么删除 WorkerId：

- TaskHub 当前是单体 Laravel 项目。
- 短期内不会多实例部署。
- 所有业务写入先由同一个应用生成 ID，不需要区分不同服务器。
- 保留 WorkerId 会让学习和维护成本变高，但当前没有实际收益。

WorkerId 什么时候才需要：

- 多台服务器同时生成 ID。
- 多个应用实例同时写入同一个数据库。
- 使用队列、异步任务、横向扩容后，不同进程之间必须避免 ID 冲突。

因此当前 TaskHub 使用简化版 Snowflake 更合适。它不是完整分布式雪花算法，但符合当前单体 MVP 架构。

### 3. 新增 POST /tasks

文件：

```text
routes/web.php
```

新增：

```php
// 发布任务在任务大厅模态框内提交，不单独创建 /tasks/create 页面。
Route::post('/tasks', [TaskController::class, 'store'])->name('tasks.store');
```

该路由放在 `EnsureSsoAuthenticated` 中间件组里，保证只有登录用户能发布任务。

### 4. 外部付款账号接口

文件：

```text
app/Integrations/Payment/PaymentAccountClient.php
app/Integrations/Payment/PaymentAccount.php
app/Integrations/Payment/PaymentAccountException.php
config/payment_account.php
```

为什么要先做这个封装：

- 付款账号是外部主数据，TaskHub 不维护本地付款账号表。
- 前端只应该提交 `paymentAccountId`，不能提交 `paymentAccountName` 作为可信数据。
- 如果让前端提交账号名称，用户可以在浏览器里改请求，导致任务里保存错误名称。
- 账号名称、部门等历史展示信息必须由后端实时查询外部接口后保存快照。
- 外部接口地址、path、请求方式都集中在配置中，Controller 不直接拼 URL。

`PaymentAccountClient` 的职责：

```text
paymentAccountId
↓
读取 config/payment_account.php
↓
使用 Laravel Http Client 请求外部付款账号接口
↓
解析 JSON 响应
↓
返回 PaymentAccount 对象
```

`PaymentAccount` 的职责：

- 表示“已经从外部接口查到的付款账号”。
- 把外部响应规整为 `accountId`、`accountName`、`departmentId`、`departmentName`。
- 通过 `toSnapshot()` 生成写入 `task.payment_account_snapshot` 的历史快照。

当前支持两种 GET 写法：

```env
# 账号 ID 放在 path 里，最终请求 /accounts/PAY001。
PAYMENT_ACCOUNT_PATH=/accounts/{accountId}
PAYMENT_ACCOUNT_METHOD=GET
```

```env
# 账号 ID 放在 query string 里，最终请求 /accounts?accountId=PAY001。
PAYMENT_ACCOUNT_PATH=/accounts
PAYMENT_ACCOUNT_METHOD=GET
```

当前也支持 POST JSON：

```env
PAYMENT_ACCOUNT_PATH=/accounts/detail
PAYMENT_ACCOUNT_METHOD=POST
```

POST 时 body 为：

```json
{
  "accountId": "PAY001"
}
```

注意：这里没有提前加入 `clientId`、`secret` 等认证字段，因为目前还没有确认付款账号接口是否需要认证。如果公司接口确认需要认证，再把配置项补到 `config/payment_account.php`，并只在 `PaymentAccountClient` 内部组装请求。

### 5. TaskController 增加 store()

文件：

```text
app/Http/Controllers/TaskController.php
```

发布任务完整流程：

1. 读取当前 SSO 用户。
2. 解析附件 ID。
3. 通过 `PaymentAccountClient` 调外部接口获取付款账号快照。
4. 开启数据库事务。
5. 生成 task ID。
6. 创建 `task`，状态直接为 `OPEN`。
7. 写入发布者 JSON 快照。
8. 写入付款账号 JSON 快照。
9. 写入 `TASK_CREATED`。
10. 写入 `TASK_PUBLISHED`。
11. 如果有附件 ID，写入 `attachment_ref`。

注意：

- 当前只做招标发布，`assignment_type = BIDDING`。
- DIRECT 指派后续单独开发，不提前塞进本章。
- 附件文件本体不进 TaskHub，只保存外部附件 ID。
- 外部账号查询发生在数据库事务之前，避免数据库事务等待网络请求。
- 如果外部付款账号接口失败，Controller 把错误挂到 `paymentAccountId` 字段，让前端表单展示校验错误。

### 6. Layout 增加 actions

文件：

```text
resources/js/Layouts/AppLayout.tsx
```

`AppLayout` 增加 `actions?: ReactNode`，用于放页面级按钮，例如“发布任务”。

这样做比在每个页面自己重写标题栏更稳：

- 标题层级一致。
- 页面右上角操作区一致。
- 后续任务详情、交付、变更也可以复用。

### 8. Index.tsx 增加发布任务表单

文件：

```text
resources/js/Pages/Tasks/Index.tsx
```

关键点：

- 使用 `useForm<TaskCreateForm>()` 管理表单数据、错误和提交状态。
- 表单字段名必须和 `StoreTaskRequest` 一致。
- 提交到 `POST /tasks`。
- 成功后关闭弹窗、清空表单。
- Laravel redirect 回任务大厅，Inertia 自动刷新列表。
- 前端只提交 `paymentAccountId`，账号名称和部门由后端调用外部付款账号接口得到。

表单字段：

| 前端字段 | 后端校验字段 | 数据库字段 |
|---|---|---|
| `title` | `title` | `task.title` |
| `description` | `description` | `task.description` |
| `budget` | `budget` | `task.budget` |
| `expectedDelivery` | `expectedDelivery` | `task.expected_delivery` |
| `biddingDeadline` | `biddingDeadline` | `task.bidding_deadline` |
| `complexity` | `complexity` | `task.complexity` |
| `paymentAccountId` | `paymentAccountId` | `task.payment_account_id` |
| `attachmentIds` | `attachmentIds` | `attachment_ref.attachment_id` |

注意：`paymentAccountName` 不再由前端提交。发布时后端根据 `paymentAccountId` 调用外部付款账号接口，得到 `accountName`、`departmentId`、`departmentName` 后写入 `payment_account_snapshot`。这是为了保证历史快照来自可信服务，而不是来自浏览器表单。

### 9. Flash 成功提示

后端：

```php
return redirect()
    ->route('tasks.index')
    ->with('success', '任务已发布。');
```

Inertia 中间件共享：

```php
'flash' => [
    // 表单提交成功后，Controller 可以通过 redirect()->with('success', ...) 给页面一次性提示。
    'success' => $request->session()->get('success'),

```

前端读取：

```tsx
const { flash } = usePage<SharedPageProps>().props;
```

## 常见错误

| 错误 | 原因 | 处理方式 |
|---|---|---|
| 419 CSRF 错误 | 表单没有走 Inertia 或 CSRF token 缺失 | 使用 Inertia `useForm().post()`，不要手写 fetch |
| `id` 不能为空 | 忘记生成 BIGINT 主键 | 确认使用 `SnowflakeId->next()` |
| 发布时间保存失败 | `biddingDeadline` 字段名和 Request 不一致 | 前端字段必须叫 `biddingDeadline` |
| 付款账号名称不可信 | 前端不应提交账号名称 | 只提交 `paymentAccountId`，后端用 `PaymentAccountClient` 查询 |
| 付款账号接口未配置 | `.env` 缺少 `PAYMENT_ACCOUNT_BASE_URL` 或 `PAYMENT_ACCOUNT_PATH` | 按公司真实接口填写配置 |
| 附件重复错误 | 同一个附件 ID 重复关联 | 后端 `attachmentIds()` 已去重 |
| 弹窗提交后不关闭 | 没有在 `onSuccess` 中关闭 | `setIsCreateOpen(false)` |
| 页面没有成功提示 | `flash` 没共享给 Inertia | 检查 `HandleInertiaRequests` |

## 本章总结

本章完成了 TaskHub 第一个写入业务闭环：

```text
React Dialog 表单
↓
Inertia useForm
↓
Laravel FormRequest
↓
TaskController 事务
↓
task / attachment_ref / task_event
↓
返回任务大厅
```
