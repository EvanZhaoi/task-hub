# 14-外部接口接入

## 本章目标

本章集中说明 TaskHub 如何接入外部系统接口。

目前 TaskHub 有三类外部接口：

| 接口 | 状态 | 用途 |
|---|---|---|
| SSO 当前登录人接口 | 已完成 | 用 `accessToken` 获取当前登录人的总部身份信息 |
| 交易/付款账号接口 | 已完成 | 获取全部账号供发布任务 Select 选择，并在保存时生成付款账号快照 |
| 本据点全员人员接口 | 已完成 | 登录时增强 Session 用户信息，未来作为开发者选择器数据源 |

本章的重点不是再开发新功能，而是把外部接口的接入规范讲清楚。以后新增外部接口时，优先沿用这一章的结构，不要把 HTTP 调用散落在 Controller、Model 或 React 页面里。

## 学习目标

读完本章后，你应该理解：

- 为什么外部接口调用要封装在 `app/Integrations`。
- 为什么 Controller 不直接调用 `Http::get()` 或 `Http::post()`。
- 为什么外部接口配置放在 `config/*.php` 和 `.env`。
- 为什么不能把 `clientSecret` 暴露到前端。
- 为什么页面展示信息可以用快照，但权限判断必须用实时或后端可信数据。
- Laravel HTTP Client 和 Spring Boot `RestTemplate` / `WebClient` 的概念对应。
- 如何用 `Http::fake()` 测试外部接口调用，而不真实访问公司服务。

## 最终效果

当前仓库形成了这样的外部接口结构：

```text
app/
└── Integrations/
    ├── Sso/
    │   ├── SsoClient.php
    │   ├── SsoException.php
    │   └── SsoUser.php
    ├── Payment/
    │   ├── PaymentAccountClient.php
    │   ├── PaymentAccountException.php
    │   └── PaymentAccount.php
    └── Personnel/
        ├── PersonnelClient.php
        ├── PersonnelException.php
        └── PersonnelUser.php

config/
├── sso.php
├── payment_account.php
└── personnel.php
```

请求流向是：

```text
Controller / Service
↓
Integration Client
↓
读取 config/*.php
↓
Laravel Http Client
↓
外部系统
↓
Value Object 解析响应
↓
Controller / Service 使用稳定对象
```

## 一、外部接口接入原则

### 1. 不在 Controller 里直接拼外部接口

不要这样写：

```php
// 不推荐：Controller 直接知道外部接口地址、参数和响应结构。
$response = Http::post('https://sso.company.com/api/current-user', [
    'accessToken' => $token,
]);
```

原因：

- Controller 会越来越胖。
- 外部接口路径、参数、响应字段一变，业务 Controller 也要跟着改。
- 测试时很难只替换外部接口调用。
- 代码里容易泄露真实地址、密钥或协议细节。

TaskHub 的做法：

```php
// 推荐：Controller 只依赖语义清楚的客户端。
$user = $ssoClient->fetchCurrentUser($accessToken);
```

这样 Controller 只关心“我要当前登录人”，不关心外部接口是 POST、GET、query string、JSON body 还是其它形式。

### 2. 每类外部接口一个目录

当前目录：

```text
app/Integrations/Sso
app/Integrations/Payment
app/Integrations/Personnel
```

命名规则：

- `Client`：负责 HTTP 请求。
- `Exception`：负责表达这一类外部接口异常。
- 值对象：负责把外部响应变成 TaskHub 内部稳定结构。

例如付款账号：

```text
PaymentAccountClient
PaymentAccountException
PaymentAccount
```

这和 Java / Spring Boot 中常见写法类似：

| Laravel | Spring Boot 类比 |
|---|---|
| `PaymentAccountClient` | `PaymentAccountClient` / `PaymentAccountGateway` |
| Laravel `Http` facade | `RestTemplate` / `WebClient` |
| `PaymentAccount` readonly object | DTO / record |
| `PaymentAccountException` | 自定义 RuntimeException |
| `config/payment_account.php` | `application.yml` 配置绑定 |

这里不使用 Repository，因为 Repository 是给数据库持久化抽象准备的。外部 HTTP 接口更适合叫 Client 或 Integration Client。

### 3. 配置写在 `.env`，代码读 `config`

`.env.example` 只放占位，不放真实公司地址和密钥：

```env
SSO_BASE_URL=
SSO_LOGIN_URL=
SSO_LOGOUT_URL=
SSO_CLIENT_ID=
SSO_CLIENT_SECRET=
SSO_USERINFO_PATH=

PAYMENT_ACCOUNT_BASE_URL=
PAYMENT_ACCOUNT_LIST_PATH=
PAYMENT_ACCOUNT_DETAIL_PATH=

PERSONNEL_BASE_URL=
PERSONNEL_LIST_PATH=

REDIS_CLIENT=phpredis
REDIS_HOST=127.0.0.1
REDIS_PORT=6379
REDIS_CACHE_DB=1
EXTERNAL_DIRECTORY_CACHE_STORE=redis
EXTERNAL_DIRECTORY_CACHE_TTL=86400
```

代码不直接读 `env()`，而是通过 `config()`：

```php
// 读取 config/sso.php 中的配置。
$baseUrl = config('sso.base_url');
```

原因：

- Laravel 正式环境通常会执行 `php artisan config:cache`。
- 配置缓存后，业务代码中直接 `env()` 可能拿不到预期值。
- 统一通过 `config()`，更符合 Laravel 官方习惯。

修改 `.env` 后执行：

```bash
# 清理配置缓存，让 Laravel 重新读取 .env。
php artisan config:clear

# 如果页面或 Blade 有缓存问题，再清理视图缓存。
php artisan view:clear
```

## 二、外部大列表缓存策略

付款账号和本据点人员列表数据量比较大，查询时间比较长，因此当前不再要求每次页面打开或每次登录都实时拉全量数据。

TaskHub 使用 Redis 缓存这两类外部大列表：

| 缓存 | 默认 key | 数据来源 |
|---|---|---|
| 付款账号列表 | `taskhub:payment_accounts` | `PaymentAccountClient::refreshCache()` |
| 本据点人员列表 | `taskhub:personnel_users` | `PersonnelClient::refreshCache()` |

### 1. 配置

```env
# 普通 Laravel Cache 仍然可以保持 file，不影响 Session 和其它基础功能。
CACHE_STORE=file

# 外部大列表单独指定 Redis store。
EXTERNAL_DIRECTORY_CACHE_STORE=redis

# 缓存 TTL，单位秒。默认 86400 秒，即 1 天。
EXTERNAL_DIRECTORY_CACHE_TTL=86400

# Redis 连接。TaskHub 当前只把 Redis 用于外部大列表缓存。
REDIS_CLIENT=phpredis
REDIS_HOST=127.0.0.1
REDIS_PASSWORD=null
REDIS_PORT=6379
REDIS_DB=0
REDIS_CACHE_DB=1

# 两类外部大列表的 cache key。
PAYMENT_ACCOUNT_CACHE_KEY=taskhub:payment_accounts
PERSONNEL_CACHE_KEY=taskhub:personnel_users
```

注意：

- `SESSION_DRIVER` 仍然是 `file`，当前不把登录 Session 放进 Redis。
- `QUEUE_CONNECTION` 仍然是 `sync`，当前不引入队列 Worker。
- Redis 只用于缓存外部大列表，避免频繁慢查询外部接口。

### 2. 读取规则

运行时读取顺序：

```text
TaskController@index / SsoController@store
↓
PaymentAccountClient::fetchAll() / PersonnelClient::fetchAll()
↓
优先读取 Redis 缓存
↓
缓存命中：直接返回缓存
缓存未命中：尝试请求外部接口并写入缓存
```

为什么缓存未命中时仍尝试请求外部接口：

- 新环境第一次部署时 Redis 可能还没有数据。
- 手动执行定时命令前，系统仍然可以尝试自愈。
- 如果外部接口失败，原来的异常处理仍然生效。

### 3. 更新规则

定时更新使用命令：

```bash
php artisan taskhub:sync-external-directories
```

命令内部会执行：

```text
PaymentAccountClient::refreshCache()
PersonnelClient::refreshCache()
```

刷新规则：

- 外部接口请求成功。
- 返回列表非空。
- 只有满足以上条件才覆盖 Redis 旧缓存。
- 如果请求失败，不清空旧缓存。
- 如果返回空列表，不清空旧缓存。

这对应你的要求：

```text
如果未获取到数据，则不更新
```

### 4. 定时调度

当前在 `routes/console.php` 中注册：

```php
// 每 30 分钟刷新一次外部大列表。
Schedule::command('taskhub:sync-external-directories')
    ->everyThirtyMinutes()
    ->withoutOverlapping();
```

服务器上还需要配置 Laravel 调度器入口：

```bash
* * * * * cd /path/to/task-hub && php artisan schedule:run >> /dev/null 2>&1
```

本地开发时可以手动执行：

```bash
php artisan taskhub:sync-external-directories
```

### 5. 为什么不把数据存 MySQL

当前不把付款账号和人员列表落到 TaskHub 数据库，原因：

- 它们是外部主数据，TaskHub 不是主数据系统。
- 当前只需要选择器和快照，不需要本地维护生命周期。
- Redis 缓存可以降低慢查询压力，又不会引入本地数据同步表的设计复杂度。
- 后续如果要做本地搜索、权限规则或审计，再评估是否需要落库。

## 三、SSO 当前登录人接口

### 1. 职责

SSO 接口只负责回答一个问题：

```text
这个 accessToken 对应哪个总部用户？
```

当前代码：

```text
app/Integrations/Sso/SsoClient.php
app/Integrations/Sso/SsoUser.php
```

### 2. 配置

```env
SSO_BASE_URL=https://sso.company.com
SSO_LOGIN_URL=https://sso.company.com/oauth/authorize
SSO_LOGOUT_URL=https://sso.company.com/logout
SSO_CLIENT_ID=ClientID
SSO_CLIENT_SECRET=secret
SSO_USERINFO_PATH=/api/current-user
SSO_TIMEOUT=3
SSO_VERIFY_SSL=false
```

注意：

- `SSO_LOGIN_URL` 和 `SSO_LOGOUT_URL` 是浏览器跳转地址，所以是完整 URL。
- `SSO_USERINFO_PATH` 是后端接口 path，只写 `/api/current-user`。
- `SSO_CLIENT_SECRET` 只能在后端使用，不能传给 React。

### 3. 调用流程

```text
React SSO Callback 页面
↓
POST /sso/session，提交 accessToken
↓
SsoController
↓
SsoClient::fetchCurrentUser($accessToken)
↓
POST JSON 到总部 SSO userinfo 接口
↓
SsoUser::fromPayload($payload)
↓
返回 SsoUser
```

### 4. 请求 body

当前按公司推荐方式发送 JSON：

```json
{
  "clientId": "ClientID",
  "secret": "secret",
  "accessToken": "token-from-sso"
}
```

对应代码在：

```text
app/Integrations/Sso/SsoClient.php
```

### 5. 响应结构

当前支持总部返回嵌套结构：

```json
{
  "id": "response-001",
  "user": {
    "employeeNo": "E10001",
    "displayName": "张三",
    "departmentId": "DEV01",
    "departmentName": "开发一部"
  }
}
```

`SsoUser::fromPayload()` 会从 `user` 里读取人员字段。

### 6. 为什么不在前端调 SSO userinfo

原因：

- 前端不能持有 `clientSecret`。
- 浏览器请求公司接口可能遇到 CORS。
- 当前用户身份必须由后端确认，不能相信前端提交的姓名、部门或角色。

正确方式：

```text
前端只提交 accessToken
↓
Laravel 后端带 clientId / secret 调总部接口
↓
后端写 Session
```

## 四、交易/付款账号接口

### 1. 职责

付款账号接口负责两件事：

1. 页面打开任务大厅时，获取所有可选账号，作为发布任务弹窗的 Select 选项。
2. 发布任务保存时，根据选中的账号 ID 获取可信账号信息，写入 `task.payment_account_snapshot`。

当前代码：

```text
app/Integrations/Payment/PaymentAccountClient.php
app/Integrations/Payment/PaymentAccount.php
```

### 2. 配置

```env
PAYMENT_ACCOUNT_BASE_URL=https://payment.company.com
PAYMENT_ACCOUNT_LIST_PATH=/api/payment-accounts
PAYMENT_ACCOUNT_DETAIL_PATH=/api/payment-accounts/{accountId}
PAYMENT_ACCOUNT_METHOD=GET
PAYMENT_ACCOUNT_TIMEOUT=3
PAYMENT_ACCOUNT_VERIFY_SSL=false
PAYMENT_ACCOUNT_CACHE_KEY=taskhub:payment_accounts
```

说明：

- `PAYMENT_ACCOUNT_LIST_PATH` 用于获取全部账号。
- `PAYMENT_ACCOUNT_DETAIL_PATH` 用于保存时按账号 ID 查询详情。
- 如果没有详情接口，可以把 `PAYMENT_ACCOUNT_DETAIL_PATH` 留空，保存时会从账号列表中按 ID 查找。

### 3. 页面 Select 流程

```text
浏览器访问 /tasks
↓
TaskController@index
↓
PaymentAccountClient::fetchAll()
↓
外部账号列表接口
↓
PaymentAccount::listFromPayload()
↓
paymentAccountOptions 作为 Inertia props 下发
↓
React Select 展示账号选项
```

前端只保存选中的 `paymentAccountId`：

```tsx
// 付款账号来自后端调用外部接口得到的列表，前端只提交选中的账号 ID。
<NativeSelect
    name="paymentAccountId"
    onChange={(event) => form.setData('paymentAccountId', event.target.value)}
    value={form.data.paymentAccountId}
>
    <option value="">请选择付款账号</option>
</NativeSelect>
```

### 4. 发布任务保存流程

```text
React 提交 paymentAccountId
↓
StoreTaskRequest 校验 paymentAccountId
↓
TaskController@store
↓
PaymentAccountClient::fetchById($paymentAccountId)
↓
得到 PaymentAccount
↓
PaymentAccount::toSnapshot()
↓
写入 task.payment_account_snapshot
```

为什么保存时还要再查一次：

- 不能相信浏览器提交的账号名称。
- 用户打开页面后，账号名称可能已经变化。
- `task.payment_account_snapshot` 是业务发生时的历史展示信息，应该来自后端可信接口。

### 5. 支持的列表响应

`PaymentAccount::listFromPayload()` 支持几种常见结构。

直接数组：

```json
[
  {
    "accountId": "PAY001",
    "accountName": "开发一部创新预算"
  }
]
```

包在 `data`：

```json
{
  "data": [
    {
      "accountId": "PAY001",
      "accountName": "开发一部创新预算"
    }
  ]
}
```

包在 `accounts`：

```json
{
  "accounts": [
    {
      "accountId": "PAY001",
      "accountName": "开发一部创新预算"
    }
  ]
}
```

## 五、本据点全员人员接口

### 1. 职责

本据点人员接口负责两件事：

1. 登录时在 Session 中追加本据点人员信息，不覆盖总部 SSO 原始人员信息。
2. 未来作为“指定开发者”“协作者选择器”的人员数据源。

当前代码：

```text
app/Integrations/Personnel/PersonnelClient.php
app/Integrations/Personnel/PersonnelUser.php
```

### 2. 配置

```env
PERSONNEL_BASE_URL=https://personnel.company.com
PERSONNEL_LIST_PATH=/api/users
PERSONNEL_METHOD=GET
PERSONNEL_TIMEOUT=3
PERSONNEL_VERIFY_SSL=false
PERSONNEL_CACHE_KEY=taskhub:personnel_users
```

### 3. 登录增强流程

```text
SsoClient 获取总部当前登录人
↓
得到总部 employeeNo
↓
PersonnelClient::findByEmployeeNo($employeeNo)
↓
读取本据点全员人员列表
↓
按工号匹配
↓
匹配成功：Session 顶层保留总部 SSO 信息，并追加 siteUser 数组
匹配失败：Session 只保留总部 SSO 人员信息
```

这样做的原因：

- 总部接口证明身份，但字段可能不够完整。
- 本据点人员列表更适合业务展示和人员选择。
- TaskHub 不需要维护本地 `users` 表。
- 非本据点人员仍然可以登录，不会因为本地列表没有记录而失败。
- 总部 SSO 信息是认证来源，不能被本据点人员列表覆盖。

匹配到本据点人员时，Session 中的 `sso_user` 类似：

```php
[
    // 顶层字段仍然来自总部 SSO 当前登录人接口。
    'employeeNo' => '00010001',
    'displayName' => '总部张三',
    'departmentId' => null,
    'departmentName' => null,
    // siteUser 是额外数组字段，保存本据点更完整的信息。
    'siteUser' => [
        'employeeNo' => '10001',
        'displayName' => '张三',
        'departmentId' => 'DEV01',
        'departmentName' => '开发一部',
    ],
]
```

### 4. 工号前导 0 处理

本据点工号不带前导 0。如果总部返回纯数字工号：

```text
00010001
```

匹配人员列表时会按本据点规则规整为：

```text
10001
```

注意：

- 只有纯数字工号会去掉前导 0。
- 带字母的工号，例如 `E001`，不会处理。
- 这样可以避免误改公司真实工号格式。

### 5. 支持的人员列表响应

直接数组：

```json
[
  {
    "employeeNo": "10001",
    "displayName": "张三",
    "departmentId": "DEV01",
    "departmentName": "开发一部"
  }
]
```

包在 `data`：

```json
{
  "data": [
    {
      "employeeNo": "10001",
      "displayName": "张三"
    }
  ]
}
```

包在 `users`：

```json
{
  "users": [
    {
      "employeeNo": "10001",
      "displayName": "张三"
    }
  ]
}
```

## 六、错误处理

### 1. SSO 错误

SSO 当前登录人接口失败时，登录不能继续。

代码行为：

```text
SsoException
↓
SsoController 返回 401 JSON
↓
React Callback 页面展示登录失败
```

原因：

- 没有 SSO 当前登录人，就不能确认用户身份。
- 不能用前端提交的姓名或工号兜底。

### 2. 本据点人员列表错误

本据点人员列表接口失败时，登录可以继续。

代码行为：

```text
PersonnelException
↓
SsoController 记录 warning 日志
↓
Session 保留总部 SSO 人员信息，不追加 siteUser
```

原因：

- SSO 已经证明用户身份。
- 人员列表只是增强展示信息。
- 如果因为人员列表短暂失败就阻止登录，会影响可用性。

### 3. 付款账号错误

任务大厅加载账号列表失败时：

```text
PaymentAccountException
↓
TaskController@index 返回空 paymentAccountOptions
↓
React Select 显示“付款账号列表不可用”
↓
禁用发布按钮
```

发布任务保存时账号查询失败：

```text
PaymentAccountException
↓
TaskController@store 转成 paymentAccountId 字段错误
↓
发布任务表单展示错误
```

原因：

- 没有可信付款账号，就不能发布任务。
- 不能让前端手填账号名称绕过校验。

## 七、测试方式

### 1. 使用 Http::fake()

Laravel 测试外部 HTTP 调用时，不真实访问公司接口。

示例：

```php
// Http::fake 会拦截 Laravel HTTP Client 请求。
// 这样测试不会依赖真实网络，也不会调用公司测试环境。
Http::fake([
    'https://payment.example.test/accounts/PAY001' => Http::response([
        'accountId' => 'PAY001',
        'accountName' => '开发一部创新预算',
    ]),
]);

$paymentAccount = app(PaymentAccountClient::class)->fetchById('PAY001');

expect($paymentAccount->accountId())->toBe('PAY001');
```

### 2. 使用容器替换 Client

测试 Controller 时，可以替换外部接口 Client：

```php
// 把真实 PaymentAccountClient 替换成测试专用匿名类。
// Controller 仍然通过依赖注入拿到 PaymentAccountClient，不需要改生产代码。
$this->app->instance(PaymentAccountClient::class, new class extends PaymentAccountClient
{
    public function fetchById(string $accountId): PaymentAccount
    {
        return new PaymentAccount(
            accountId: $accountId,
            accountName: '开发一部创新预算',
        );
    }
});
```

这和 Spring Boot 测试里用 `@MockBean` 替换外部 Client 的思路类似。

## 八、常见错误

| 错误 | 原因 | 处理方式 |
|---|---|---|
| `base URL is not configured` | `.env` 没配置对应 `*_BASE_URL` | 填写 `.env` 后执行 `php artisan config:clear` |
| `path must be a path, not a full URL` | path 配置写成了完整 URL | `BASE_URL` 写域名，`*_PATH` 只写 `/api/...` |
| 登录成功但部门不准确 | 本据点人员列表没匹配到当前工号 | 检查总部工号和本据点工号格式，特别是前导 0 |
| 任务大厅付款账号为空 | 付款账号列表接口未配置或请求失败 | 检查 `PAYMENT_ACCOUNT_BASE_URL` 和 `PAYMENT_ACCOUNT_LIST_PATH` |
| 发布任务账号报错 | 保存时无法从外部接口找到选中账号 | 检查账号列表和详情接口返回的 `accountId` 是否一致 |
| 测试真实请求了公司接口 | 忘记 `Http::fake()` 或容器替换 Client | Feature Test 中必须 fake 外部接口 |
| 修改 `.env` 后仍旧报旧配置 | Laravel 配置缓存未清理 | 执行 `php artisan config:clear` |

## 九、新增外部接口模板

以后如果要接入新的外部接口，按这个模板做。

### 1. 新增配置文件

```text
config/example_service.php
```

示例：

```php
<?php

return [
    // 外部服务基础地址，只放域名和公共前缀。
    'base_url' => env('EXAMPLE_SERVICE_BASE_URL'),
    // 具体接口 path，只写 path，不写完整 URL。
    'path' => env('EXAMPLE_SERVICE_PATH'),
    // 请求超时时间。
    'timeout' => (int) env('EXAMPLE_SERVICE_TIMEOUT', 3),
    // 是否校验 HTTPS 证书。
    'verify_ssl' => filter_var(env('EXAMPLE_SERVICE_VERIFY_SSL', false), FILTER_VALIDATE_BOOLEAN),
];
```

### 2. 新增 Integration 目录

```text
app/Integrations/ExampleService/
├── ExampleServiceClient.php
├── ExampleServiceException.php
└── ExampleResult.php
```

### 3. Controller 只依赖 Client

```php
public function store(ExampleServiceClient $client)
{
    // Controller 调用语义方法，不直接拼 HTTP 请求。
    $result = $client->fetchSomething();
}
```

### 4. 测试必须 fake 外部接口

```php
Http::fake([
    'https://example.test/api/*' => Http::response([
        'id' => '001',
    ]),
]);
```

## 本章总结

TaskHub 外部接口接入的核心规范是：

```text
外部接口调用集中放在 app/Integrations
↓
配置放在 config/*.php 和 .env
↓
Controller 只调用 Client 的语义方法
↓
响应结构由值对象解析
↓
测试使用 Http::fake 或容器替换
```

当前三个外部接口的边界：

- SSO：证明当前登录人是谁。
- 付款账号：提供发布任务时可选的交易账号，并生成历史快照。
- 本据点人员列表：补全当前用户 Session 信息，未来提供人员选择器数据。
