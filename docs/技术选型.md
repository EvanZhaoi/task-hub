## 1. 选型原则

| 原则 | 说明 |
|---|---|
| **沿用成熟栈** | 前端统一使用 React 19 + TypeScript + Vite；后端沿用 Spring Boot 3 + MySQL 8 + Redis + Docker 的成熟组合 |
| **少即是多** | MVP 不引入新语言、新架构；能用标准库的不引依赖；能用云服务的不自建 |
| **接口先行** | 对外接口（人员信息、附件存储）必须用抽象层封装，方便后续替换 |
| **可演示可部署** | 选型必须支持 Mac mini 本地开发 + 单一 Docker Compose 一键起 |

## 3. 前端选型详解

### 3.1 为什么是 React 19

| 维度 | React 19 | 说明 |
|---|---|---|
| 生态 | 最大 | shadcn/ui、Radix、TipTap、React Router、Zustand 都是一线组合 |
| 招聘 / 协作 | 友好 | 前端协作者更容易接手 |
| 组件模型 | 函数组件 + Hooks | 与当前主流 React 项目一致 |
| 现有项目 | 已初始化 React 前端 | 后续重写沿用 React，不再回切旧框架 |

> **结论**：**React 19**。本项目后续前端重写以 React + TypeScript 为唯一口径。

### 3.2 为什么用 TypeScript

- 大型项目必备
- 提前发现类型错误，减少 runtime bug
- IDE 自动补全
- 招聘 / 协作友好

### 3.3 UI 库：shadcn/ui + Radix UI

**锁定**：**shadcn/ui**（MIT）+ 底层 **Radix UI**（MIT）

**为什么选 shadcn/ui**：
- **MIT 开源 + 可商用**
- **复制粘贴模式**：组件代码复制到 `src/components/ui/`，不是 npm 依赖——不会被升级绑架
- **纯 Tailwind 底层**：与我们 Tailwind 配置天然契合，可直接覆盖品牌色 #5e6ad2
- **Radix UI 提供可访问性**：Dialog、Select、Dropdown 等"难做对"的组件由 Radix 管 a11y / 焦点 / 键盘
- **现代美感**：默认 shadcn 风格，干净专业

**架构**：
```
src/components/ui/      ← shadcn/ui 组件（你的代码）
├── button.tsx          ← cva 变体 + cn() 工具
├── card.tsx
├── input.tsx
├── dialog.tsx          ← 包装 Radix Dialog
└── ... (按需加)

src/lib/utils.ts        ← cn() 工具（clsx + tailwind-merge）
```

**添加新组件**（以后）：
```bash
npx shadcn@latest add select  # 自动加 Select 组件
npx shadcn@latest add tabs    # 加 Tabs
```

**使用**：
```tsx
<Button variant="destructive" size="sm">删除</Button>
<Dialog open={showModal} onOpenChange={setShowModal}>
  <DialogContent>
    <DialogTitle>确认删除</DialogTitle>
    <p>确定要删除吗？</p>
  </DialogContent>
</Dialog>
```

**不在 shadcn 内的（项目自写）**：
- TaskCard（业务组件）
- BidListItem（业务组件）
- GanttChart（甘特图，特殊交互）
- StatusBadge（状态徽章，定制）
- AppHeader（应用布局）

### 3.4 Gantt：自写 vs 库

| 选项 | MVP 适用度 | 长期 |
|---|---|---|
| **自写 CSS + HTML**（推荐） | ✅ 够用 | 拖拽 / 依赖关系时不够 |
| dhtmlx-gantt | ❌ 商业授权（$$$） | — |
| gantt-task-react | ✅ 免费 | 二期可评估 |

> **结论**：**自写**。MVP 只需"展示"不需"编辑"。二期如要拖拽改交期，再评估 dhtmlx 替代品。

## 4. 后端选型详解

### 4.1 为什么是 Spring Boot 3 + Java 21

- Evan 10 年 Java 经验，零学习成本
- Spring Boot 3 是 Jakarta EE 10 时代标准
- Java 21 LTS 性能优于 17（虚拟线程已 GA）
- 生态成熟，文档全

### 4.2 ORM：MyBatis-Plus vs Spring Data JPA

| 维度 | MyBatis-Plus | Spring Data JPA |
|---|---|---|
| SQL 可控性 | ✅ 自己写 | ⚠️ 框架生成，难调优 |
| 复杂查询 | ✅ XML / 注解 | ⚠️ Specification / QueryDSL 复杂 |
| 单表 CRUD | ✅ BaseMapper 自动 | ✅ Repository 自动 |
| 性能 | 略优 | 略逊（反射多） |
| 学习曲线 | ⭐⭐ | ⭐⭐⭐ |
| 团队倾向 | 多数国内团队 | 多数外企 |

> **结论**：**MyBatis-Plus**（推荐）。Evan 需拍板确认。**待决策 3**。

### 4.3 后端包结构

```
com.evanzhaoi.taskhub
├── config/          # 配置类（跨域、安全、Swagger）
├── controller/      # REST API
├── service/         # 业务逻辑
│   └── impl/
├── mapper/          # MyBatis Mapper（如果用 MyBatis-Plus）
├── entity/          # 数据库实体
├── dto/             # 请求/响应 DTO
├── enums/           # 枚举（任务状态等）
├── exception/       # 自定义异常
├── common/          # 工具类
├── integration/     # 外部系统对接（人员信息 API、附件接口）
│   ├── personnel/   # 外部人员接口
│   └── attachment/  # 外部附件接口
└── TaskHubApplication.java
```

### 4.4 错误处理

- 全局 `@RestControllerAdvice` 统一捕获
- 错误码 + 错误信息 + 业务数据（如有）
- 前端 axios 拦截器统一弹 toast

### 4.5 集成架构（公司 SSO + 人员信息接口）

TaskHub 本身**不实现登录**，通过两个外部接口拿身份信息。

```
业务方调 TaskHub
     │
     ▼
[Request Header: Authorization: Bearer <token>]
     │
     ▼
┌─────────────────────────────────┐
│   Spring 拦截器 / 过滤器          │
│   - 提取 token                   │
│   - 调 SSO 验证接口 / 验签         │
│   - 拿到 employeeNo（工号）       │
│   - 查 Redis 缓存                │
└────────────┬────────────────────┘
             │ 未缓存
             ▼
   调公司人员信息接口
   GET /api/personnel/{employeeNo}
   拿到 name/dept/avatar/email
   写 Redis（TTL 5-10 分钟）
             │
             ▼
   继续 Spring MVC 流程
   Controller 能拿到 UserContext
```

**包结构**：
```
com.evanzhaoi.taskhub.integration
├── sso/
│   ├── SsoClient.java            # 调公司 SSO 验证接口
│   ├── TokenValidator.java       # token 验证接口
│   └── dto/                      # 请求/响应 DTO
├── personnel/
│   ├── PersonnelClient.java      # 调公司人员信息接口
│   ├── UserContext.java          # 缓存中的用户信息
│   └── dto/
└── cache/
    └── UserContextCache.java     # Redis 缓存封装
```

**关键设计原则**：
- 业务代码只依赖 `UserContext`，不直接调外部接口
- 外部接口调用都走抽象层（`SsoClient` / `PersonnelClient`），后续换协议不动业务
- 缓存优先，未命中才调外部

## 5. 数据库 & 存储

### 5.1 MySQL 8.4 LTS

- 已用过的稳定版本
- 支持 JSON 列（存富文本 HTML 不用建子表）
- 全文索引（未来做搜索升级用）

### 5.2 Redis 7.x 用途

| 用途 | 存什么 | TTL |
|---|---|---|
| Session | 用户登录态 | 7 天 |
| 验证码 | 短信 / 邮件验证码 | 5 分钟 |
| 限流 | 接口调用计数 | 1 分钟 |
| 任务搜索热词 | 搜索历史 | 30 天 |
| 分布式锁 | 选标操作防并发 | 10 秒 |

> MVP 用单一 Redis 实例（与后端同机器），集群按需。

### 5.3 附件集成

**MVP**：复用已有附件上传 / 下载接口。

- TaskHub 不保存文件内容
- TaskHub 不实现上传 / 下载
- TaskHub 只保存外部附件 ID，并通过 `attachment_ref` 关联到任务或投标
- 附件展示需要文件名、大小、下载地址时，由后端按外部附件 ID 调用附件接口获取

```java
public interface AttachmentClient {
    AttachmentInfo getInfo(String attachmentId);
    String getDownloadUrl(String attachmentId);
}
```

### 5.4 数据库设计原则

- 主键：雪花 ID（long，分布式友好）
- 删除策略：MVP 不做软删除字段，核心业务数据通过状态和事件保留历史
- 审计：`created_at`, `created_by`, `updated_at`, `updated_by`
- 状态：枚举 + 注释（前端要同步）
- JSON 字段：仅用于变更前后值等结构化快照；协作成员使用关系表
- 索引：状态、created_at、payment_account_id、created_by

## 6. 关键开放决策（需 Evan 拍板）

> 以下 5 个决策对系统形态有重大影响，必须先定。其他技术细节可开发过程中再定。

### 决策 1：鉴权方式 ✅ 已定（公司插件）

**结论**：TaskHub **不做登录、不发 token、不管密码**。统一接入公司现有 SSO 插件。

- 业务方登录 → 公司 SSO 插件颁发 token
- 业务方调 TaskHub API → 请求头带 `Authorization: Bearer <token>`
- TaskHub 验证 token 有效性 → 拿到工号 → 调公司人员信息接口拿详细资料
- 业务代码只关心"当前 user 是谁"，不关心怎么登录

**具体方案**（与 Evan 确认后细化）：
- **token 验证**：调公司 SSO 验证接口（1 次 HTTP）或本地验证 JWT 签名（二期）
- **人员信息**：独立调用人员信息接口（与验证接口可能同源也可能分开）
- **结果缓存**：Redis 缓存用户信息 5-10 分钟（避免每次请求都调外部）
- **撤销**：依赖 SSO 插件自身机制，TaskHub 不额外处理

**接口需求清单**（需 Evan 提供）：
- [ ] SSO 验证端点 URL + 协议（REST / gRPC）
- [ ] token 格式示例（JWT? opaque?）
- [ ] 人员信息接口是否返回部门、邮箱、头像等
- [ ] 错误码约定（token 过期 / 无效 / 权限不足）

> 实现代码隔离在 `integration/sso/` 与 `integration/personnel/` 两个包，业务代码不依赖具体协议。

### 决策 2：富文本编辑器 ✅ 已定

**锁定**：**A. TipTap**

**选型理由**：
- 基于 ProseMirror（业界标准），React 集成成熟
- 扩展性强：@ 提及 / 表格 / 协作编辑等二期功能易加
- 社区活跃，文档中文友好

**协议核对**（2026-06-11 确认）：
- TipTap core：MIT
- ProseMirror：MIT
- StarterKit / Link / Placeholder 等官方扩展：MIT
- **✅ 开源 + 可商用**

**必用扩展**（v0.3 锁定的功能矩阵）：
- `@tiptap/starter-kit`（粗体 / 斜体 / 下划线 / 删除线 / 标题 / 列表）
- `@tiptap/extension-link`（链接）
- `@tiptap/extension-placeholder`（占位符）

**二期可加**：
- `@tiptap/extension-mention`（@ 提及人）
- `@tiptap/extension-table`（表格）
- `@tiptap/extension-image`（图片）

### 决策 3：ORM 选型 ✅ 已定

**锁定**：**A. MyBatis-Plus**（Apache 2.0）

**选型理由**：
- SQL 可控：任务搜索 LIKE + 复合索引能写得很精准
- 报告 / 统计查询方便
- 国内社区成熟，问题好查
- BaseMapper 自动覆盖单表 CRUD

**关键依赖**：
- `mybatis-plus-spring-boot3-starter`（适配 Spring Boot 3）
- `mybatis-plus-jsqlparser`（复杂 SQL 解析）
- 分页插件 `PaginationInnerInterceptor`

**避坑**：
- 多租户等复杂场景慎用插件
- 复杂 SQL 仍写 XML 注释清楚

### 决策 4：任务调度 ✅ 已定

**锁定**：**A. Spring `@Scheduled`**（MVP） + **C. XXL-JOB**（二期）

**MVP 场景**：
- 每天 00:30 扫描 `OPEN` 且已截止、并且不存在 `ACTIVE` 投标的任务 → 自动 `FAILED`
- 有 `ACTIVE` 投标的截止任务保持 `OPEN`，页面派生展示为“待选标”
- MVP 不实现软删除清理任务

**实现位置**：
- `@EnableScheduling` 放在主启动类
- 调度方法放 `com.evanzhaoi.taskhub.scheduler` 包
- 配合 `@Async` 避免阻塞调度线程

**二期升级**（XXL-JOB）：
- 分布式调度
- 任务失败重试
- 可视化运维

### 决策 5：部署目标 ✅ 已定

**锁定**：**A. Mac mini 本地**（开发/演示） + **B. 云 ECS**（生产）

**架构路径**：
1. **开发/演示阶段**（0 → 内部试运行）
   - Mac mini 本地 Docker Compose
   - Cloudflare Tunnel 暴露公网（已有 Playbook）
   - 域名 `<project>.demo.<rootdomain>`
2. **生产阶段**（试运行 → 全员上线）
   - 迁到云 ECS（2C4G 起步，按量扩容）
   - MySQL 用云 RDS（减少运维）
   - Redis 用云托管（减少运维）
   - 附件仍走外部附件系统，TaskHub 只保存附件 ID

**Docker Compose 服务清单**（本地开发）：
- `taskhub-backend`（Spring Boot 应用）
- `taskhub-frontend`（Nginx + React 静态资源）
- `mysql`（8.4 LTS）
- `redis`（7.x）
- `adminer`（数据库管理，可选）

**生产环境待决**（二期）：
- 镜像仓库：阿里云 ACR / Docker Hub
- 日志：云日志服务（SLS）
- 监控：云监控 + Spring Actuator

## 7. 暂时搁置（开发过程中定）

| 主题 | 搁置原因 |
|---|---|
| 消息队列 | MVP 无高并发 |
| 全文搜索（ES） | 任务量 < 1k，MySQL LIKE 够用 |
| 分布式锁 | 单机部署够用 |
| 链路追踪（Sleuth / SkyWalking） | MVP 无分布式 |
| Prometheus + Grafana | MVP 无大规模监控 |
| 对象存储 | 由外部附件系统负责，TaskHub 不自建 |
| WebSocket（实时通知） | 通知 MVP 用轮询，WS 二期 |
| 多租户 | 内部系统，单租户足够 |
| 国际化（i18n） | 全员中文 |
| 暗黑模式 | DESIGN.md 暂不支持 |
| token 验证优化 | MVP 调外部接口，二期可加本地 JWT 验签 |
