## 2026-06-12 · 任务复杂度字段（v0.5 重定义）

**操作人**：Harry
**协作方**：Evan（产品负责人）

### 上下文
上午 Evan 让我加"难易程度"字段（简单/普通/困难），v0.5 上线。13:10 Evan 反领：把概念从"**难易程度**"重新定义为"**复杂程度**"，值从"简单/普通/困难"改成"**简单 / 中等复杂 / 高度复杂**"。

为避免"字段名 difficulty + 文案 复杂程度"的语义割裂，**同时重命名**：

| 维度 | 上午版（已废弃） | 当前版（v0.5 锁定） |
|---|---|---|
| 概念名 | 难易程度 | **复杂程度** |
| 字段名 | `difficulty` | `complexity` |
| 英文枚举 | `EASY / NORMAL / HARD` | `LOW / MEDIUM / HIGH` |
| 中文标签 | 简单 / 普通 / 困难 | **简单 / 中等复杂 / 高度复杂** |
| 默认值 | NORMAL（普通） | MEDIUM（中等复杂） |
| 组件 | `DifficultyBadge` | `ComplexityBadge` |
| CSS 类 | `.badge-easy/-normal/-hard` | `.badge-low/-medium/-high` |

### 做了什么
- **DB** (`schema.sql`)：列名 `difficulty` → `complexity`，注释 + 索引名同步改
- **类型** (`api/types.ts`)：`Difficulty` → `Complexity`，值改为 `LOW/MEDIUM/HIGH`
- **组件**：删 `DifficultyBadge.tsx`，新建 `ComplexityBadge.tsx`
- **CSS** (`globals.css`)：三个 badge 类重命名
- **视图**：`TaskCard` / `TaskDetailView` / `CreateTaskView` / `HomeView` 全部重命名引用 + 中文文案
- **状态** (`stores/tasks.ts`)：`filterDifficulty` / `setDifficultyFilter` → `filterComplexity` / `setComplexityFilter`
- **mock 数据**：8 条任务改为新枚举值（CRM 屎山=HIGH、订单导出=LOW 等）
- **文档**：
  - 需求文档 v0.5：§6.1 字段描述、§13 L16、§11 MVP 必须做、版本历史都更新
  - 模块与数据库 v0.7：4.2.1 task 表、4.4 索引策略、版本历史都更新
  - `DESIGN.md` §2.3：表头 + 行名 + 值都改
  - 本交接记录段落直接覆盖上午那段

### 关键决策
- **字段名跟概念名一致**：避免后端开发者读 `difficulty` 字段但代码注释/UI 都写"复杂度"的认知错位
- **枚举 LOW/MEDIUM/HIGH 替代 EASY/NORMAL/HARD**：跟"复杂程度"语义贴合（"低复杂度"比"简单"更准确）
- **默认值 = MEDIUM**：跟"未标 = 中等"的直觉对齐
- **不影响定价**：MVP 只作标签 + 筛选，二期再考虑接定价/接单意愿排序
- **历史数据**：还未真上线，mock 全部同步改；真上线时如已有数据需一次性 UPDATE

- 2026-06-12 12:57：v0.5 需求 + v0.7 模块设计 + schema.sql + 前端全套同步升级（难度版）
- 2026-06-12 13:10：v0.5 重定义 → 复杂程度 + LOW/MEDIUM/HIGH，全套再同步升级

### 下一步
